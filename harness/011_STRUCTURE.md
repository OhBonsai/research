### C11：形式化规约语言（Formal Specification Language）⬆️新增类别

**因果条件**：Harness实践分散，缺乏统一的形式化描述
**现象**：不同团队对Harness的定义、组成、边界理解不一致

**归纳推导**：NLAH论文是首篇对Agent Harness进行形式化的学术工作，其提出的六要素（Contracts/Roles/Stage Structure/Adapters/State Semantics/Failure Taxonomy）提供了一套可被不同实现共享的描述语言。这不是一个"实现机制"，而是一个"描述框架"——它描述的是 Harness 应该包含什么，而不是如何实现。

**行动策略**：
- **Contracts**：输入输出规约+格式约束+验证门+权限边界+重试/停止规则（NL-01）
- **Roles**：显式角色定义，职责不重叠（NL-02）
- **Stage Structure**：显式工作负载拓扑（NL-03）
- **Adapters/Scripts**：命名确定性钩子（NL-04）
- **State Semantics**：外部化+路径可寻址+压缩稳定（NL-05）
- **Failure Taxonomy**：命名故障模式驱动恢复策略（NL-06）
- **IHR架构**：Intelligent Harness Runtime（NL论文）
- **Harness三层递进定义**：Prompt(说什么)→Context(给什么)→Harness(什么条件下运行)（CN-02, Louis Bouchard）

**聚合标签**：NL-01/02/03/04/05/06/07/08, CN-02

### C12：评估与基准测试（Evaluation & Benchmarking）⬆️新增类别

**因果条件**：Harness工程效果需要严格的实验验证
**现象**：缺乏标准化评估方法，不同Harness间的比较缺少统一基准

**归纳推导**：评估方法从 C9（可观测性）中独立出来的逻辑是——C9 关注"运行时监控"（production observability），而 C12 关注"离线评估"（offline evaluation）。State of Agent Engineering 报告明确指出两者之间存在显著缺口（89% vs 37%），证明它们是不同的工程实践。

**行动策略**：
- **pass@k vs pass^k**：能力上限 vs 一致性可靠性（EV-01）
- **Task/Trial/Grader框架**：模块化评估组件（EV-02）
- **三类评分器**：代码/模型/人工（EV-03）
- **轨迹vs结果区分**：独立评估关注点（EV-04）
- **策略遵从独立度量**：100%完成但33%策略遵从（EV-05）
- **四支柱评估模型**：LLM/Memory/Tool/Environment（EV-06）
- **LLM-Judge vs Agent-Judge成本对比**：16倍成本差异（EV-07）
- **Agent崩溃模式**：螺旋进入怪异行为而非渐进退化（EV-08）
- **pass^k方差现象**：跨尝试一致性退化（EV-09）
- **长程可靠性测试**：覆盖50-100+工具调用（EV-10）
- **HAL统一评估CLI**：框架无关的跨基准测试（EV-11）
- **轨迹加密防污染**：防止评估数据泄露到训练集（EV-12）

## 第三部分：选择性编码（Selective Coding）

### 3.1 核心类别（更新）

**核心类别**：**有限上下文窗口约束下的可靠性环境工程**

（Reliability Environment Engineering Under Bounded Context Window Constraints）

**更新后的统摄力检验**：

12 个类别在核心类别下的归属逻辑：
- C1-C2（上下文+记忆）：直接管理窗口约束本身
- C3-C4（约束+验证）：确保窗口内行为可靠性
- C5（编排隔离）：通过分治缓解窗口限制
- C6（自演化）：维持长期可靠性
- C7（可拆卸性）：保障环境工程的可维护性
- C8（人机协作）：定义顶层治理框架
- C9（可观测性）：提供运行时度量数据
- **C10（企业安全）**：可靠性在企业场景下的安全延伸——Agent行为可靠性不仅指功能正确，还包括安全合规 ✅统摄
- **C11（形式化规约）**：为环境工程提供统一的描述语言——描述"什么是好的环境设计" ✅统摄
- **C12（评估基准）**：量化验证环境工程的效果——证明Harness确实提升了可靠性 ✅统摄

### 3.2 故事线（更新版）

> LLM 具有强大的推理能力，但运行在根本性受限的上下文窗口中。Harness Engineering 的回应不是让模型看得更多，而是**设计一个让模型在有限视野内依然可靠工作的环境**。
>
> 这个环境有十二个工程维度：**分层记忆**（C2）将知识持久化为文件，使Agent不再每次从零开始；**上下文生命周期管理**（C1）通过 Reduce/Offload/Isolate 三原则精确控制窗口内信息。**架构约束**（C3）用代码和 Contracts 替代 prompt 软约束；**验证管道**（C4）从 PreCompletion 到 Failure Taxonomy，在多点检查拦截错误。**多Agent编排**（C5）通过 Roles/Stage Structure 分治大任务，每个执行者从干净上下文开始。
>
> **自演化与熵治理**（C6）确保系统不退化——通过 Trace→Critic→Refiner 闭环和数据飞轮效应，执行轨迹甚至成为模型改进的训练数据。**可拆卸性**（C7）使 Harness 不绑定特定模型（Post-Training 耦合现象证明 Harness 对性能的影响可能超过模型选择本身）。
>
> **人机协作**（C8）定义了人的新角色——从编码者到环境设计者，这一转变在控制论中有238年的历史同构。**可观测性**（C9）提供运行时数据基础，但与**评估基准**（C12）是不同的实践——89%有观测但仅37%有在线评估。pass@k衡量能力上限，pass^k衡量一致性可靠性，两者的张力定义了Harness优化的目标空间。
>
> 在企业场景下，**安全与合规**（C10）成为必须——58%有监控但仅37%有终止开关的治理-遏制缺口，以及间接数据访问绕过等新攻击面，使安全成为Harness不可缺的维度。**形式化规约**（C11）则为整个领域提供了统一的描述语言——NLAH的六要素框架使不同实现之间的对话成为可能。

### 6.4 理论饱和判断

初版从互联网文献中识别出C10(企业安全)/C11(形式化规约)/C12(评估基准)三个新类别。经审查，C11(形式化规约)的NLAH六要素已被C2-C5等类别分散吸收，作为独立类别属"描述视角"而非"实现维度"，予以去除。C7(可拆卸性)本质为架构决策，并入C3。最终精简为**10个核心类别**。

# 012_SYSTEM: Harness Engineering 系统要素模型

> 本文档为 011_STRUCTURE 扎根理论编码与结构检验的精炼产出。
> 材料基础：3篇内部文章 + 30+篇互联网文献，167个概念标签 → 10个核心类别。

## 一、核心命题

**Agent = Model + Harness**

Harness 是模型之外的一切：提示词、工具与技能描述、执行环境、编排逻辑、中间件与策略。

**核心类别**：有限上下文窗口约束下的可靠性环境工程

LLM 推理能力强大，但运行在根本性受限的上下文窗口中。Harness Engineering 不是让模型更聪明，而是设计一个让模型在有限视野内依然可靠工作的环境。这一核心命题统摄以下全部 10 个要素类别。

**类比框架**：Model = CPU，Context Window = RAM，Harness = OS，Agent = Application

## 二、十大要素体系

### C1 上下文生命周期管理（Context Lifecycle Management）

**本质**：管控窗口内信息的质量与数量，使模型始终在最佳利用率区间工作。

**核心机制**：
- **生命周期四阶段**：注入 → 监控 → 压缩 → 归档
- **40% 利用率阈值**：超过此值推理质量显著下降；上下文接近极限时性能下滑 50-70%
- **Reduce / Offload / Isolate 三原则**：压缩上下文、卸载到外部存储、多Agent隔离
- **渐进式披露**：AGENTS.md 作为导航地图（~100行），详细内容按需加载
- **结构化上下文**：XML标签/分隔符组织上下文块，信息优先级分层
- **焦点锚定**：TODO List 即使空操作也保持任务焦点
- **Prompt缓存优化**：静态内容前置 + 变量内容后置，最大化前缀命中
- **推理三明治**：xhigh(规划) → high(实现) → xhigh(验证)，按阶段分配算力
- **硬性截断**：CLAUDE.md 首200行/25KB，系统指令 32KiB 上限

**反模式**：上下文死亡螺旋——错误→更多上下文→决策下降→更多错误→循环不可逆

---

### C2 分层记忆架构（Layered Memory Architecture）

**本质**：跨会话持久化知识，使 Agent 具备积累和演化能力。

**核心机制**：
- **三类记忆**：
  - 程序记忆（Procedural）= 行为规则（AGENTS.md / rules.md）
  - 情景记忆（Episodic）= 历史经验（Session / Checkpoint）
  - 语义记忆（Semantic）= 知识库（domain/*.md）
- **Everything is a File**：四类记忆全部文件化，可读/可写/可版本追踪——自演化的基础条件
- **State Semantics 三性质**：外部化、路径可寻址、压缩稳定
- **可插拔文件系统后端**：统一存储抽象（目录/文件读写/glob/grep/shell），支持多种实现
- **配置层级优先级**：$HOME → 项目根 → 当前目录，层级覆盖
- **置信度标注**：区分 [hot-path, unverified] 与 [user-confirmed]

**目录结构范式**：
```
agent-root/
├── AGENTS.md           ← 唯一总入口，~100行目录
├── memory/
│   ├── working/        ← 当前任务状态
│   ├── procedural/     ← 行为规则 + 用户偏好
│   ├── semantic/       ← 知识地图 + 领域知识
│   └── episodic/       ← 会话记录 + 检查点
└── evolution/          ← 待审核 + GC日志 + 冲突
```

**核心约束**：Agent 在运行时无法访问的东西就不存在。

### C3 架构约束与模块化（Architectural Constraint & Modularity）

**本质**：用工程手段替代 prompt 劝说，将规则从"希望被遵守"变为"不遵守就报错"；同时确保 Harness 不绑定特定模型，适配 AI 模型快速迭代的现实。

**约束机制**：
- **仓库是Agent的OS**：一切编码到仓库，不在仓库里Agent就看不见
- **层级编号系统**：L0(类型) → L1(工具) → L2(配置) → L3(业务) → L4(接口)，高层可引低层反之不行
- **中心化约束 + 本地自治**：只管架构边界，内部实现自由
- **Contracts（合约）**：输入输出规约 + 格式约束 + 验证门 + 权限边界 + 重试/停止规则
- **工具白名单 + 动态注册**：只暴露安全工具集合，按任务类型注册
- **参数强类型约束**：Pydantic/TypeScript 类型系统拒绝非法输入
- **权限四级分层**：默认 → 自动编辑 → 计划模式 → 自动模式（渐进授权）
- **幂等性设计**：重复执行不产生副作用，Agent 可安全重试
- **沙箱隔离 + 最小权限暴露**：当前任务最小所需工具集
- **配置驱动行为**：持久化机器可读配置（AGENTS.md / CLAUDE.md）引导行为

**模块化机制**（原C7可拆卸性）：
- **三层架构分离**：应用层 / Harness核心层 / 模型适配层
- **抽象模型接口**：统一模型调用，屏蔽 API 差异
- **工具定义模型无关**：工具描述与 function calling 格式解耦
- **Prompt模板分离**：模板外置为可配置资源
- **能力特性标记**：标记模型是否支持视觉/长上下文/函数调用等

**关键数据**：
- 格式即性能乘数——Hashline 单一格式修改使性能从 6.7% 升至 68.3%
- Post-Training 耦合——同一模型 Opus 4.6 在不同 Harness 中排名 #33 vs #5，Harness 影响可能超过模型选择
- 三阶段工程演进：Prompt Engineering(2022-24) → Context Engineering(2025) → Harness Engineering(2026)

**场景偏向警示**：当前 Harness 实践偏向 Web 开发（LLM训练数据饱和领域），嵌入式系统/小众语言/非编码场景回报可能递减。

### C4 验证管道与自愈（Validation Pipeline & Self-Healing）

**本质**：机械化检查替代 LLM"直觉"，将错误拦截前移。

**核心机制**：
- **事前预验证**：写代码前先检查合法性（2次交互替代10次tool call）
- **四层验证管道**：build(编译) → lint-arch(架构合规) → test(功能正确) → verify(端到端)
- **四类验证点**：前置条件 / 步骤后 / 进展检测(状态指纹比对) / 终止边界(预算式)
- **三级错误升级**：自我校正 → 换策略重试 → 升级人工 → 彻底中止+诊断报告
- **PreCompletionChecklist**：拦截退出强制对照任务规约验证
- **LoopDetection**：追踪每文件编辑次数，N次后强制换方法
- **Failure Taxonomy**：命名故障模式（缺失制品/路径错误/验证失败/工具错误/超时）驱动恢复策略
- **Ralph Loop**：拦截退出 → 重注入目标 → 利用文件系统跨窗口持续执行
- **回压验证**：静默成功 + 大声失败 = 上下文高效的反馈循环
- **四阶段会话分离**：Research → Plan → Execute → Verify

**理论基础**：验证-生成不对称性（P vs NP 直觉）——验证答案通常比生成答案容易。

**关键数据**：有效验证机制可将输出质量提升 2-3 倍。

### C5 多Agent编排与隔离（Multi-Agent Orchestration & Isolation）

**本质**：通过分治策略将大任务拆解为多个小上下文任务，每个执行者从干净状态开始。

**核心机制**：
- **协调者绝不写代码**（中等以上任务最重要的原则）
- **两层架构**：协调者(规划/委派/汇总) + 执行者(干净上下文/精确prompt)
- **复杂度分级委派**：一句话→直接做；需清单→委派；需设计决策→委派+隔离
- **模型分级调度**：Haiku(简单) / Opus(复杂) / Flash(检索)——成本降 60-70%
- **Roles 显式定义**：solver / verifier / researcher / orchestrator，职责不重叠
- **Stage Structure**：显式工作负载拓扑 plan → execute → verify → repair
- **四重隔离**：
  - 任务边界隔离（全新上下文，只传必要定义）
  - 信息接口化（结构化消息替代原始上下文共享）
  - 错误隔离（一个Agent错误不传播到下游）
  - Git Worktree隔离（结构性变更在临时副本执行）
- **交叉模型Review**：不同模型review避免思维盲区重叠
- **跨产品共享Agent Loop**：CLI/Web/IDE/App 复用同一套执行逻辑

**关键数据**：75%+ 组织在生产中使用多模型路由。

### C6 自演化与熵治理（Self-Evolution & Entropy Governance）

**本质**：对抗长期退化，实现系统越运行越高效。

**核心机制**：
- **三种演化机制**：
  - 被动（hot-path）：用户纠正/任务完成时写入，标注[unverified]
  - 主动（显式）：用户说"记住"时直接写目标文件，标注[user-confirmed]
  - 后台（GC）：定时扫描，矛盾升级人工裁决
- **自演化闭环**：Agent执行 → 验证抓到问题 → Critic分析模式 → Refiner更新规则 → 下一个Agent受益
- **GC 四目标**：矛盾内容 / 过时决策 / 知识空白 / 漂移模式
- **轨迹编译**：同类任务成功3次以上 → 编译为确定性脚本（如 `make add-endpoint`）
- **棘轮效应**：编译的模式成为永久基础设施，Agent 被释放去处理新问题
- **Review → Lint规则**：反复出现的review问题编码为新lint规则——软知识变硬规则
- **上下文蒸馏**：保留决策结论，丢弃推理过程
- **数据飞轮**：执行轨迹即训练数据，部署即训练，竞争优势从模型参数迁移到轨迹捕获
- **模型-Harness共进化**：社区摸索 → Post-training → 模型内化 → 新Harness支持更高阶能力
- **Harness Debt**：Harness本身也需维护，类比技术债务

**规律**：Bitter Lesson 在应用层的体现——架构假设 6 个月内过时，需拥抱持续重构。Manus 6个月重构 5 次 Harness。

### C7 人机协作模型（Human-Agent Collaboration Model）

**本质**：定义人在 Agent 环境中的角色——从编码者转为环境设计者。

**核心机制**：
- **harness-creator / harness-executor** 分工：creator 搭基础设施，executor 在其中执行任务
- **项目健康度评分**：0-20裸奔，21-70有缺口，71+健康
- **执行计划审批**：非简单任务生成执行计划文件，人扫一眼批准
- **门控人工干预**：interrupt_on 参数将特定操作映射到审批门
- **三级错误升级**含人工裁决环节
- **问题驱动，非过度预防**：聚焦已发生的问题，由实际失败驱动迭代
- **配置起点原则**：从简单开始，仅在观察到失败后增加配置
- **渐进复杂度路径**：六阶段渐进建设，从 Context 到 Detachability

**理论基础**：控制论同构——瓦特离心调速器(1788) → Kubernetes声明式控制 → AI Agent Harness，238年同一模式：当传感器和执行器足够强大时，工程师从直接操作转向系统设计。

**关键发现**：Brooks 定律未触发——团队 3→7 人未出现沟通爆炸，因为 Agent 间依赖是环境级而非代码级。

### C8 可观测性与质量度量（Observability & Quality Metrics）

**本质**：提供 Harness 运行时监控的数据基础。

**核心机制**：
- **六维可观测性**：上下文健康 / 执行效率 / 验证质量 / 上下文治理 / 工具可靠性 / 端到端性能
- **六钩子中间件架构**：before_agent / before_model / wrap_model / wrap_tool / after_model / after_agent
- **Trace-Based 主动反馈**：Agent 获取自身执行轨迹并派生分析
- **工具质量 > 工具数量**：5个清晰工具优于20个模糊工具
- **错误信息教学质量**：好的报错=一次教学（什么违反了/为什么/怎么修）
- **自主决策审计**：审计须捕获 Agent 推理"为什么"而非仅"做了什么"
- **证据质量审计轨迹**：SOC 2 合规需业务上下文 + 取证重建能力

**关键缺口**：可观测性 ≠ 评估——89% 有可观测性但仅 37% 有在线评估（见 C10）。

---

### C9 企业安全与合规治理（Enterprise Security & Compliance）

**本质**：Agent 行为可靠性在企业场景下的安全延伸。

**核心机制**：
- **机器主体身份**：Agent 作为自主"主体"需要与人类同等的身份认证/授权
- **治理-遏制缺口**：58-59% 有监控但仅 37-40% 有终止开关——可见性 ≠ 可干预性
- **间接数据访问绕过**：Agent 通过数据库查询/配置文件读取绕过传统 DLP
- **Prompt级DLP**：在查询时、网络出口前检测和脱敏敏感信息
- **工具级访问控制**：限制特定 MCP 工具调用而非文件系统级权限
- **用途绑定**：约束 Agent 到预期功能，超越权限范围
- **影子AI发现**：未授权 Agent(浏览器扩展/个人账户/本地安装) = 不可控攻击面
- **伦理墙**：防止 Agent 跨敏感业务功能滥用（类比金融信息隔离墙）
- **通道级安全管理**：按通信通道(邮件/API/数据库)设定独立策略
- **默认行为即治理**：策略编码为默认行为而非反应式过滤

**关键数据**：100% 任务完成但仅 33% 策略遵从——安全合规与功能正确是独立维度。

### C10 评估与基准测试（Evaluation & Benchmarking）

**本质**：离线量化验证 Harness 工程效果（区别于 C9 的运行时监控）。

**核心机制**：
- **pass@k vs pass^k**：
  - pass@k = k次中至少1次成功（衡量能力上限）
  - pass^k = k次全部成功（衡量一致性可靠性，如单次75% → 三次全成功仅42%）
- **Task/Trial/Grader 框架**：模块化评估组件（任务定义 + 试验执行 + 评分逻辑）
- **三类评分器**：代码型(快/脆) / 模型型(灵活/非确定) / 人工(金标准/不可扩展)
- **轨迹 vs 结果区分**：交互历史与最终环境状态是独立评估关注点
- **策略遵从独立度量**：任务完成率与策略遵从率须分开测量
- **四支柱评估模型**：LLM组件 / 记忆系统 / 工具行为 / 环境
- **消融实验方法**：逐模块递增添加，测量差分贡献
- **长程可靠性测试**：覆盖 50-100+ 工具调用，1%榜单差异无法检测第50步后的漂移
- **Agent 崩溃模式**：螺旋进入"怪异行为"而非渐进退化
- **HAL 统一评估 CLI**：框架无关的跨基准测试（SWE-bench/USACO/AppWorld/OSWorld等9+）
- **轨迹加密防污染**：加密存储 Agent 轨迹防止评估数据泄露到训练集

**关键数据**：
- LLM-as-Judge $0.06 vs Agent-as-Judge $0.96（16倍成本差异）
- Harness 优化 52.8%→66.5%（+13.7pp，不换模型）

## 三、要素间结构关系

```
C1(上下文管理) ←─ 约束 ─→ C3(架构约束与模块化) ←─ 扩展 ─→ C9(企业安全)
      ↑                          ↑                            │
      │ 管理                规则来源                      合规要求
      ↓                          │                            ↓
C2(分层记忆) ←─ 存储 ─→ C6(自演化/熵治理) ─── 轨迹 ───→ C10(评估基准)
      ↑                          │                            │
      │ 支撑                进化产物                      度量反馈
      ↓                          ↓                            ↓
C5(多Agent编排) ←─ 验证 ─→ C4(验证管道) ←── 数据 ──→ C8(可观测性)
      ↑
      │ 协作
      ↓
C7(人机协作)
```

**关系逻辑**：
1. C1↔C3：上下文管理依赖架构约束限制信息流入；约束本身也是上下文的一部分
2. C2↔C6：记忆是熵治理的对象；熵治理产物（清洁后的知识）回写到记忆
3. C5→C1：多Agent编排的核心动机就是管理上下文（协调者不写代码以保护上下文）
4. C6→C4：自演化将Review中的软知识编译为验证管道的硬规则
5. C9→C3：企业安全是架构约束在合规维度的延伸
6. C10↔C6：评估数据驱动演化；演化产生的轨迹成为评估输入
7. C3内部：模块化（原C7）确保约束体系不绑定特定模型实现

## 四、VSM 可行系统映射

| VSM子系统 | 功能 | Harness要素 |
|-----------|------|-------------|
| **S1 操作** | 执行基本工作 | C3(工具/沙箱/适配层) + C4(验证) |
| **S2 协调** | 操作间冲突处理 | C5(隔离) + C1(上下文管理) |
| **S3 控制** | 优化效能/分配资源 | C5(协调者) + C7(人机协作) |
| **S3* 审计** | 深入检查操作实际 | C4(验证管道) + C8(可观测) + C10(评估) |
| **S4 智能** | 扫描环境/学习改进 | C6(自演化) + C2(记忆) + C10(评估) |
| **S5 政策** | 定义身份与价值观 | C7(人机协作) + C3(约束) + C9(安全) |

**Ashby 必要多样性定律验证**：Harness 通过多层互补策略匹配 Agent 行为多样性——硬约束(C3)压缩行为空间，验证管道(C4)检测逸出，隔离(C5)防止扩散，熵治理(C6)防止长期积累，人工升级(C8)兜底。"预防→检测→纠正→升级"的层叠式多样性匹配。

## 五、关键量化证据

| 数据点 | 含义 |
|--------|------|
| Harness优化 52.8%→66.5%（不换模型） | Harness工程直接效果 |
| Hashline格式改变 6.7%→68.3% | 微小环境变化产生巨大性能差异 |

## 六、场景迁移关注点（Coding → Office）

当前 Harness 方法论主要从 Coding Agent 实践中涌现。迁移到企业办公场景时，以下维度需重点关注：

| 关注点 | 说明 |
|--------|------|
| 用户意图消歧 | 办公指令比编码指令模糊度更高，需要更强的意图理解和澄清机制 |
| 格式多样性适配 | PDF/DOCX/PPTX/XLSX 等格式处理远超代码文件 |
| 框架场景偏向 | 当前偏 Web 开发（LLM训练数据饱和领域），Office 场景训练数据可能不饱和 |
| 多租户Harness隔离 | 不同部门/用户的 Agent 策略和数据须隔离 |

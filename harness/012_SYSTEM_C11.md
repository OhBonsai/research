# 012_SYSTEM_C11：形式化规约语言详解

> **阅读前提**：本文是对 012_SYSTEM.md 中 C11（形式化规约语言）的展开论述。
> 012_SYSTEM 中的 C11 仅用 20 行描述了 NLAH 六要素和三层递进，本文将用**类比、场景和数据**把每个概念讲透。

---

## 一、为什么需要"形式化规约"？—— 问题的由来

### 1.1 一个现实困境

假设你管理 10 万台云桌面，现在要让 3 个团队分别为"合同审批""报表生成""IT 运维"场景各做一个 AI Agent。半年后你发现：

- 团队 A 把所有指令塞进一个巨大的 System Prompt，称之为"Agent 配置"
- 团队 B 用 JSON 定义了工具列表和权限，称之为"Agent 框架"
- 团队 C 写了一套 Python 编排脚本，称之为"Agent 引擎"

三个团队都说自己做了"Harness"，但他们的 Harness 长得完全不一样，互相看不懂，更不可能复用。

**这就是缺乏形式化规约的代价**：每个人对"Agent 环境应该包含什么"没有共同语言。

### 1.2 类比：建筑行业的图纸标准

建筑行业解决过同样的问题。100 年前，每个建筑师画图纸的方式都不同——有人画透视图，有人画平面图，标注方式五花八门。后来行业制定了**建筑制图标准**（如 GB/T 50001）：规定了图纸必须包含哪些视图、用什么符号、怎么标注尺寸。

这个标准**不告诉你怎么设计房子**，只告诉你**用什么语言描述你的设计**。

C11 形式化规约语言在 Harness 工程中扮演的角色完全相同：**它不是一种实现方案，而是一套描述框架——告诉所有人 Harness 应该包含哪些要素、每个要素的边界是什么。**

### 1.3 逻辑推导

```
前提 1：Harness 实践分散在不同团队/框架/论文中（事实，见 011_STRUCTURE 的 167 个标签来自 30+ 来源）
前提 2：不同来源对 Harness 的定义、组成、边界理解不一致（事实，见上述三团队困境）
前提 3：工程协作需要共同语言（工程学公理）
───────────────────────
结论：需要一套形式化的描述框架，统一"Harness 由什么构成"的回答
```

## 二、三层递进：从 Prompt 到 Harness 的认知升级

在进入 NLAH 六要素之前，先理解一个重要的认知框架——**三层递进定义**。这三层不是三个独立的东西，而是同一件事（让 Agent 可靠工作）在不同抽象层次上的表达。

| 层次 | 核心问题 | 产品经理的理解 | 日常类比 |
|------|----------|---------------|----------|
| **Prompt Engineering** | "对模型**说什么**" | 写好一条指令 | 给实习生发一条微信消息 |
| **Context Engineering** | "**给模型什么信息**" | 组织好所有背景资料 | 给实习生准备一个完整的入职文件包 |
| **Harness Engineering** | "整个系统在**什么条件下运转**" | 设计整个工作环境和流程 | 设计实习生的工位、权限、审批流、导师制、考核标准 |

### 为什么要走到第三层？

**归纳推导**（从数据到规律）：

| 实验 | 操作 | 效果 | 说明了什么 |
|------|------|------|-----------|
| Hashline 实验 | 只改输出格式（`行号:哈希\|内容`） | 6.7% → 68.3% | 改"说什么"就能提升 10 倍 |
| LangChain 实验 | 优化上下文组织（不换模型） | 52.8% → 66.5% | 改"给什么"再提升 13.7pp |
| Opus 排名实验 | 同模型不同 Harness | #33 → #5 | 改"运转条件"效果超过换模型 |
| NLAH 实验 | 完整 Harness 框架（代码→文本） | 30.4% → 47.2% | 系统化 Harness 提升 16.8pp |

**规律**：每上一层，影响面更广，效果更持久，但也更复杂。**形式化规约正是第三层的描述语言。**

## 三、NLAH 六要素——"Harness 的建筑图纸标准"

NLAH（Natural-Language Agent Harness）是 2025 年发表的**首篇对 Agent Harness 进行形式化定义的学术论文**。它回答了一个关键问题：**一个完整的 Harness 至少应该包含哪六个方面的规约？**

下面逐一讲解，每个要素都附带：定义 → 类比 → 编码场景原例 → 办公场景映射。

---

### 3.1 Contracts（合约）

**定义**：对 Agent 的输入、输出、权限、重试和停止条件的显式约定。

**类比**：合同条款。你和外包公司签合同，不会只说"帮我做个系统"，而是会明确：交付物是什么格式（输出规约）、需要我提供什么（输入规约）、你能动哪些服务器（权限边界）、出了 bug 可以改几次（重试规则）、什么情况下终止合同（停止条件）。

**Contracts 包含 5 个子项**：

| 子项 | 含义 | 编码场景示例 | 办公场景映射 |
|------|------|-------------|-------------|
| 输入规约 | Agent 接收什么 | "输入为 GitHub Issue 文本" | "输入为 OA 审批单 JSON" |
| 输出规约 | Agent 产出什么格式 | "输出为 Git Diff 补丁" | "输出为审批意见 + 结论（通过/驳回/转审）" |
| 格式约束 | 输出的结构化要求 | "Diff 必须能被 `git apply` 解析" | "审批意见必须包含依据条款编号" |
| 权限边界 | Agent 能操作什么 | "只能修改 src/ 目录" | "只能读取本部门合同，不可跨部门" |
| 重试/停止规则 | 失败如何处理 | "编译失败最多重试 3 次，超时 5 分钟终止" | "审批超时 2 小时升级人工，驳回 2 次终止" |

**要点**：Contracts 不是"建议"，是**硬约束**。没有 Contracts 的 Agent 就像没有合同的外包——什么都可能发生。

---

### 3.2 Roles（角色）

**定义**：对 Agent 在不同阶段承担的职责的显式声明，职责不重叠。

**类比**：组织架构中的岗位说明书。一个人不能既是出纳又是会计（职责分离原则）。同样，一个 Agent 在验证自己产出的时候，不应该使用生成时的同一个"身份"——否则就是"自己批自己的作业"。

**NLAH 定义了四种基本角色**：

| 角色 | 职责 | 编码场景 | 办公场景映射 |
|------|------|---------|-------------|
| **Solver** | 解决问题、产出结果 | 写代码修 Bug | 起草合同条款 |
| **Verifier** | 验证结果是否合格 | 运行测试、检查编译 | 合规审查、法务审核 |
| **Researcher** | 收集所需信息 | 检索代码库、读文档 | 检索历史合同、查政策法规 |
| **Orchestrator** | 协调多个角色的调度 | 分配子任务、合并结果 | 审批流程编排、任务分派 |

**关键原则——职责不重叠**：Solver 写完的东西必须交给 Verifier 检查，不能自己检查自己。这直接对应 012_SYSTEM 中 C5（多 Agent 编排）的"协调者绝不写代码"原则。

**为什么职责分离重要？** 因为大模型有一个已知弱点：**对自己生成内容的审查能力显著低于对他人内容的审查能力**。用不同角色（甚至不同模型）做生成和验证，是工程层面对这一弱点的补偿。

---

### 3.3 Stage Structure（阶段拓扑）

**定义**：Agent 完成任务的显式工作流拓扑——先做什么、后做什么、什么条件下回退。

**类比**：工厂流水线。一辆车的生产不是"一口气从头做到尾"，而是：冲压 → 焊接 → 涂装 → 总装 → 质检，每个阶段有明确的输入输出和转移条件。如果质检不通过，回到焊接而不是重新冲压。

**NLAH 的典型阶段拓扑**：

```
plan → execute → verify → repair（如果 verify 失败）
  ↑                           │
  └───── 如果 repair 也失败 ──┘  （最多 N 次循环）
```

**办公场景映射——合同审批流**：

```
需求收集 → 条款起草 → 法务审查 → 修改（如果审查不通过）
    ↑                              │
    └──── 如果修改 3 次仍不通过 ───┘  （升级部门负责人）
```

**为什么需要显式定义阶段？** 因为如果不定义，Agent 会"自由发挥"——可能跳过验证直接输出，可能在失败后无限循环重试，可能在不该回退的地方回退。**Stage Structure 把隐式的"希望 Agent 这么做"变成了显式的"系统强制这么走"。**

---

### 3.4 Adapters / Scripts（适配器 / 脚本）

**定义**：连接 Agent 与外部环境的确定性钩子——不依赖模型推理，用传统代码完成。

**类比**：办公室里的打印机驱动。你不需要 AI 来理解"把文档发送到打印机"，这是一个确定性操作，用驱动程序就行。Adapters 就是 Harness 中那些**不需要也不应该用模型来做的确定性操作**。

**四种典型 Adapter**：

| 类型 | 作用 | 编码场景 | 办公场景映射 |
|------|------|---------|-------------|
| **测试适配器** | 运行测试、返回结果 | `pytest`、`npm test` | 运行合规规则检查脚本 |
| **验证适配器** | 检查输出格式是否合规 | JSON Schema 验证 | 检查审批单必填字段是否完整 |
| **检索适配器** | 从知识库获取信息 | 代码库搜索（grep） | 从文档管理系统检索历史合同 |
| **解析适配器** | 将外部数据转为 Agent 可用格式 | 解析 Git log | 解析 OA 系统返回的审批历史 XML |

**关键洞察**：Adapter 是 Harness 可靠性的"锚点"。Agent 的推理过程有随机性，但 Adapter 是确定性的——它要么通过，要么不通过，没有"大概通过"。**每增加一个 Adapter，就多了一个确定性检查点，系统的可靠性就上升一个台阶。**

这直接呼应 012_SYSTEM 中 C4（验证管道）的数据："有效验证提升质量 2-3 倍"。

---

### 3.5 State Semantics（状态语义）

**定义**：Agent 工作过程中所有状态信息的存储和访问规则。

这是六要素中**最抽象但可能最重要**的一个。

**类比**：想象你在指导一个每 15 分钟就会失忆的员工。每次他忘记之前，你需要：
1. 把他做到哪儿了、做了什么决定，全写在一张纸上（**外部化**）
2. 这张纸放在一个他能找到的固定位置（**路径可寻址**）
3. 当纸太多时，能压缩成摘要但不丢失关键信息（**压缩稳定**）

这就是 State Semantics 的三个性质：

| 性质 | 含义 | 为什么重要 | 办公场景体现 |
|------|------|-----------|-------------|
| **外部化** | 状态不在模型"脑子里"，在文件/数据库里 | 模型随时可能被替换或重启，内部状态会丢失 | 审批进度存在 OA 数据库，不存在某个 Agent 的会话里 |
| **路径可寻址** | 每条状态有唯一的访问路径 | Agent 需要精确地"找到"某条历史状态 | `/{部门}/{项目}/{合同ID}/审批历史.json` |
| **压缩稳定** | 压缩后不改变语义 | 上下文窗口有限，长期运行必须压缩历史 | 把 50 轮审批讨论压缩为"关键分歧点摘要"而不丢失争议焦点 |

**为什么"状态外部化"如此关键？**

NLAH 论文的消融实验给出了直接证据：

> **在 OSWorld 基准上，移除文件支持的状态管理（file-backed state）导致性能下降 5.5 个百分点**——这是所有单因素消融中影响最大的。

这说明：**即使其他五个要素都做了，如果状态管理没做好，效果也会打折扣。**

原因在于：大模型的上下文窗口是有限的（目前主流 128K-200K tokens）。一个复杂任务的完整状态可能远超这个窗口。如果状态全靠"记在上下文里"，那么窗口用满之后就会开始遗忘。而"外部化 + 路径可寻址"意味着 Agent 可以在需要时精确调取历史状态，而不是把所有历史都塞在上下文里。

**办公场景的直接映射**：

企业办公 Agent 处理的往往是**长流程任务**（一个合同审批可能跨越数天、涉及十几个人的意见）。如果状态不外部化，每次 Agent 重新加载都要重新理解全部历史，效率极低且容易出错。State Semantics 规约要求：

1. 每个任务的状态持久化在文件/数据库中（外部化）
2. 用确定性路径访问（`/workspace/tasks/{task_id}/state.json`）
3. 历史可以被安全压缩（只保留决策点和关键结论，丢弃中间讨论细节）

### 3.6 Failure Taxonomy（故障分类）

**定义**：对 Agent 可能遇到的故障模式的命名分类，每种故障对应一条明确的恢复路径。

**类比**：医院的急诊分诊台。不是所有病人都送抢救室——感冒去普通门诊，骨折去外科，心梗才去抢救室。**分类决定处理方式。**

**NLAH 定义了 5 种基本故障模式**：

| 故障模式 | 含义 | 恢复策略 | 办公场景示例 |
|---------|------|---------|-------------|
| **Missing Artifact**（缺失制品） | Agent 需要但找不到某个文件/数据 | 触发 Researcher 角色去检索 | 审批需要的附件未上传 → 通知发起人补充 |
| **Wrong Path**（路径错误） | Agent 走错了执行路径 | 回退到上一个 Stage 重新规划 | 合同发错了审批线 → 退回重新选择审批流程 |
| **Verifier Failure**（验证失败） | 输出未通过验证检查 | 进入 repair 阶段修复 | 合同条款不符合法规 → 返回修改 |
| **Tool Error**（工具错误） | 外部工具调用失败 | 重试或换用备选工具 | OA 系统接口超时 → 重试或走离线流程 |
| **Timeout**（超时） | 某阶段超出时间限制 | 终止并升级人工处理 | 审批超 48 小时 → 自动升级上级 |

**为什么故障分类如此重要？**

没有故障分类的 Agent 遇到错误时只会做两件事：**要么无限重试，要么直接放弃**。两者都不是企业场景可以接受的。

有了故障分类，系统可以做出**差异化的、合理的响应**：
- 缺文件？去找文件，而不是重试同一个操作
- 验证失败？进入修复流程，而不是重新从头做
- 超时？升级人工，而不是继续等

**这本质上是把"异常处理"从模型的"临场判断"变成了系统的"预设规则"——用确定性代替随机性。**

## 四、IHR：从规约到运行时

NLAH 定义了"Harness 应该包含什么"（静态规约），IHR（Intelligent Harness Runtime）则定义了"规约如何运行起来"（动态运行时）。

### 4.1 IHR 三组件

| 组件 | 职责 | 类比 |
|------|------|------|
| **In-Loop LLM** | 运行时内嵌的大模型，负责推理 | 流水线上的工人（有判断力的执行者） |
| **Backend Services** | 提供工具调用、文件访问、状态存储 | 工厂的基础设施（水电气、物流系统） |
| **Runtime Charter** | 运行时生效的规则总纲（= Contracts + Roles 的实例化） | 工厂的作业指导书（当天的排产计划） |

### 4.2 NLAH 与 IHR 的关系

```
NLAH（设计时规约）           IHR（运行时实例）
─────────────────          ─────────────────
Contracts  ──实例化为──→    Runtime Charter 的一部分
Roles      ──实例化为──→    Runtime Charter 的角色指令
Stage Structure ──驱动──→   IHR 的阶段调度逻辑
Adapters   ──注册到──→     Backend Services 的工具列表
State Semantics ──实现为──→ Backend Services 的状态存储
Failure Taxonomy ──编码到──→ IHR 的异常处理分支
```

**简单说**：NLAH 是图纸，IHR 是根据图纸建成的工厂。图纸规定了"要有验证工位"，工厂里就真的有一个验证工位在运行。

## 五、消融实验：哪些要素最重要？

NLAH 论文通过**消融实验**（逐个移除要素，观察性能变化）给出了各要素的实际贡献：

### 5.1 OSWorld 基准（桌面操作任务）上的消融结果

| 消融项 | 性能变化 | 解读 |
|--------|---------|------|
| 移除 file-backed state（状态外部化） | **-5.5pp** | 状态管理是最关键的单一要素 |
| 移除 self-evolution（自演化） | -4.8pp | 学习能力的缺失影响显著 |
| 移除 multi-candidate search（多候选搜索） | +2.4pp | 在特定场景下多候选反而添乱 |

### 5.2 对产品设计的启示

**归纳推导**：

```
数据：状态外部化的消融影响（-5.5pp）> 自演化的消融影响（-4.8pp）
数据：多候选搜索在桌面任务上反而降低性能（+2.4pp 意味着移除后更好）
───────────────────────
推论 1：在企业办公场景（类似桌面操作），State Semantics 应优先实现
推论 2：并非所有 NLAH 要素都在所有场景下有正收益——需要场景化裁剪
推论 3：自演化（C6）是第二重要的投资方向
```

## 六、典型工作空间结构

NLAH 论文给出了一个**典型的 Harness 工作空间目录结构**，这是六要素在文件系统中的物化：

```
workspace/
├── AGENTS.md              ← Contracts + Roles 的入口文档（~100 行地图）
├── charter/
│   ├── contracts.md       ← 输入输出规约、权限边界、重试规则
│   ├── roles.md           ← 角色定义与职责划分
│   └── stages.md          ← 阶段拓扑定义
├── adapters/
│   ├── test_runner.py     ← 测试适配器（确定性）
│   ├── format_checker.py  ← 格式验证适配器
│   ├── retriever.py       ← 检索适配器
│   └── parser.py          ← 解析适配器
├── state/
│   ├── current.json       ← 当前任务状态（外部化、路径可寻址）
│   ├── history/           ← 历史状态存档（可压缩）
│   └── checkpoints/       ← 关键决策点快照
├── memory/
│   ├── episodic/          ← 情景记忆（事件教训）
│   ├── procedural/        ← 程序记忆（成功步骤）
│   └── failures/          ← 失败记忆（供 Critic 分析）
└── evolution/
    ├── traces/            ← 执行轨迹记录
    ├── compiled/          ← 编译后的确定性脚本
    └── rules/             ← 从 Review 中提炼的规则
```

## 七、NLAH 六要素与 012_SYSTEM 十二类别的映射关系

NLAH 六要素是**描述语言**，012_SYSTEM 的 C1-C12 是**实现维度**。两者的关系是"规约←→实现"。

| NLAH 要素 | 描述的是什么 | 对应的实现类别 |
|-----------|------------|---------------|
| Contracts | Agent 的输入输出和边界 | C3（架构约束）、C10（安全合规） |
| Roles | Agent 的职责分工 | C5（多 Agent 编排）、C8（人机协作） |
| Stage Structure | Agent 的工作流程 | C4（验证管道）、C5（多 Agent 编排） |
| Adapters | Agent 与环境的接口 | C7（可拆卸性）、C9（可观测性） |
| State Semantics | Agent 的状态管理 | C1（上下文管理）、C2（记忆架构） |
| Failure Taxonomy | Agent 的异常处理 | C4（验证管道/自愈）、C6（熵治理） |

**C11 本身不对应任何单一实现类别——它横跨所有类别，为它们提供统一的描述维度。** 这就是为什么在 012_SYSTEM 的要素关系图中，C11 的箭头指向"全部要素"。

## 八、对企业办公 Agent 产品的启示

### 8.1 直接可用的设计原则

基于上述分析，归纳 5 条直接可用于产品设计的原则：

| # | 原则 | 来源 | 产品决策 |
|---|------|------|---------|
| 1 | **State First**：状态外部化优先实现 | 消融实验：-5.5pp 最大单因素 | 第一版 MVP 必须包含文件/数据库支持的状态管理 |
| 2 | **Contract Driven**：所有 Agent 必须有显式合约 | NLAH 六要素 + 三团队困境 | 每个 Agent 上线前必须填写"合约卡片"（输入/输出/权限/重试） |
| 3 | **Role Separation**：生成和验证分离 | 角色不重叠原则 + 模型自审弱点 | 至少用两个独立调用做"生成+检查"，不能一个 Agent 自检 |
| 4 | **Adapter as Anchor**：尽可能多地用确定性适配器 | "每增加一个 Adapter 可靠性上升一个台阶" | 合规检查、格式校验、权限验证全部用确定性代码，不用模型判断 |
| 5 | **Failure Taxonomy as Day 1**：故障分类从第一天就建 | 没有分类 = 无限重试或直接放弃 | 产品需求文档中必须定义每类故障的处理策略 |

### 8.2 场景化裁剪的必要性

消融实验还提醒我们：**多候选搜索在桌面任务上反而降低性能（+2.4pp）**。这意味着不是所有 NLAH 要素在所有场景下都有正收益。

在企业办公场景中，可能需要裁剪的地方：
- **高频简单任务**（如日报生成）：可能不需要完整的 Stage Structure，一个 plan→execute 就够
- **低风险任务**（如会议纪要整理）：Failure Taxonomy 可以简化为"重试 1 次 → 升级人工"
- **高安全任务**（如财务审批）：Contracts 的权限边界和审计要求需要大幅加强

**结论**：NLAH 六要素是"完备集"，具体产品需要根据场景做**减法**（简单场景）或**加法**（高安全场景）。

> 设计依据：Factory AI probe-based evaluation [29]、Letta Context-Bench [31]、Jason Liu trajectory analysis [30]、ACON paired trajectory [12]

## 设计规范

### Case 结构

每个 case 包含以下字段：

| 字段 | 说明 |
|------|------|
| **ID** | `{类型}-{编号}`，如 `SHORT-01` |
| **场景** | 一句话描述任务目标 |
| **测试算法** | 该 case 主要验证哪些算法 |
| **种子数据** | 任务开始前需要准备的文件和数据 |
| **用户指令序列** | 按轮次列出的用户输入（模拟真实交互） |
| **Probe 问题** | 任务完成后用于评估信息保留的问题（参考 Factory AI 四类 probe） |
| **成功标准** | 可自动判定的通过条件 |
| **度量指标** | 该 case 需要记录的观测数据 |

### Probe 类型（参考 Factory AI [29]）

- **recall**：事实性保留——"最初的需求是什么？"
- **artifact**：工件追踪——"修改了哪些文件？"
- **plan**：任务规划——"下一步应该做什么？"
- **decision**：推理链保留——"为什么选择了方案 A？"

### 评分方法

- Probe 评分：LLM 盲测，6 维度各 0-5 分（准确性、上下文意识、工件追踪、完整性、连贯性、指令遵循）
- 任务完成：自动化判定（文件存在性、内容匹配、格式校验）
- 效率指标：从观测 Plugin 日志自动提取

### 执行要求

- 每个 case 跑 3 次（控制 LLM 随机性）
- 每次使用相同的种子数据（从 `eval_seeds/` 目录加载）
- 对照组：Baseline（无 Plugin）vs Full（全部算法）vs 消融组

### SHORT-01：会议纪要整理

**场景：** 将一份原始会议录音转写文本整理为结构化会议纪要。

**测试算法：** 算法 5（回压——大量文本输入的截断控制）、算法 4（缓存布局）

**种子数据：**
```
eval_seeds/short-01/
├── transcript.txt          # 会议转写文本（约 8000 字，含口语化表达、重复、跑题）
└── template.md             # 会议纪要模板（参会人、议题、决议、待办）
```

`transcript.txt` 内容要点（程序化生成）：
- 5 名参会人（虚构姓名），讨论 3 个议题
- 议题 1：Q2 预算分配（决议：市场部增加 15%，研发部维持不变）
- 议题 2：新产品上线时间（决议：推迟到 7 月 15 日）
- 议题 3：客户投诉处理流程（决议：成立专项小组，张明负责）
- 穿插大量无关闲聊（约占 40%）

**用户指令序列：**
1. "读取 transcript.txt，按 template.md 的格式整理成会议纪要"
2. "补充每个待办事项的负责人和截止日期"
3. "把决议部分用加粗标记"
4. "检查一下有没有遗漏的行动项"
5. "保存为 meeting_notes.md"

**Probe 问题：**
- recall："Q2 预算分配的具体决议是什么？"（预期：市场部增加 15%，研发部维持不变）
- artifact："最终生成了什么文件？模板中的哪些字段被填充了？"
- decision："哪些内容被判定为无关闲聊而被过滤掉了？"

**成功标准：**
- `meeting_notes.md` 存在且包含 3 个议题的决议
- 每个待办事项有负责人和截止日期
- 决议部分有 `**` 加粗标记
- 无关闲聊内容未出现在纪要中

**度量指标：**
- 总 token 消耗（对比 Baseline 时 transcript.txt 全文注入 vs Plugin 截断后的差异）
- 工具输出截断次数（算法 5）
- 缓存命中率（算法 4）

### SHORT-02：报告格式化

**场景：** 将一份纯文本数据报告转换为带图表描述的 Markdown 格式报告。

**测试算法：** 算法 1（渐进式披露——需要加载格式化相关 skill）、算法 5（回压）

**种子数据：**
```
eval_seeds/short-02/
├── raw_report.txt          # 纯文本报告（销售数据，约 3000 字）
├── chart_spec.yaml         # 图表规格要求（3 个图表的 x/y 轴定义）
└── brand_guide.md          # 品牌风格指南（标题格式、颜色代码、字体要求）
```

`raw_report.txt` 核心数据：
- 6 个月的销售数据（1月-6月），4 个产品线
- 区域分布：华东、华南、华北、西南
- 同比增长率、环比变化

**用户指令序列：**
1. "读取 raw_report.txt，转换成 Markdown 格式"
2. "按 chart_spec.yaml 的要求描述需要生成的 3 个图表"
3. "应用 brand_guide.md 的标题格式"
4. "在报告末尾添加关键洞察摘要（3 条）"
5. "保存为 formatted_report.md"

**Probe 问题：**
- recall："华东区 3 月份的销售额是多少？"
- artifact："报告中描述了几个图表？分别对应 chart_spec.yaml 中的哪些定义？"
- plan："如果要进一步细化这份报告，你会建议补充什么？"

**成功标准：**
- `formatted_report.md` 包含所有 6 个月、4 条产品线的数据
- 包含 3 个图表描述段落
- 标题格式符合 brand_guide.md
- 末尾有 3 条关键洞察

**度量指标：**
- skill 加载次数（算法 1——是否按需加载了格式化相关 skill）
- 总 token 消耗
- 工具调用次数

### SHORT-03：审批流程查询

**场景：** 在多份制度文档中查找特定审批流程的完整路径。

**测试算法：** 算法 1（渐进式披露——多文档按需加载）、算法 4（缓存布局）

**种子数据：**
```
eval_seeds/short-03/
├── policies/
│   ├── travel_policy.md        # 差旅报销制度（约 2000 字）
│   ├── procurement_policy.md   # 采购审批制度（约 3000 字）
│   ├── hr_policy.md            # 人事审批制度（约 2500 字）
│   └── finance_policy.md       # 财务审批制度（约 2000 字）
└── org_chart.yaml              # 组织架构（含审批权限层级）
```

**用户指令序列：**
1. "我要采购一批价值 8 万元的服务器，需要走什么审批流程？"
2. "审批需要哪些材料？"
3. "如果金额超过 10 万，流程有什么不同？"
4. "帮我列出完整的审批链路（从发起到最终批准）"
5. "把审批链路保存为 approval_flow.md"

**Probe 问题：**
- recall："8 万元采购需要几级审批？"
- decision："8 万和 10 万的审批流程具体差异在哪里？"
- artifact："approval_flow.md 中包含了哪些审批节点？"

**成功标准：**
- 正确识别 8 万元对应的审批层级（从 procurement_policy.md 提取）
- 正确区分 8 万和 10 万的流程差异
- `approval_flow.md` 包含完整审批链路

**度量指标：**
- 文件加载顺序和次数（算法 1——是否只加载了相关的 policy 文件）
- 缓存命中率（算法 4——多轮查询同一文档时是否命中）

### SHORT-04：邮件草稿撰写

**场景：** 根据项目进展信息撰写一封给客户的项目更新邮件。

**测试算法：** 算法 5（回压——项目文档输出的精简）、算法 4（缓存布局）

**种子数据：**
```
eval_seeds/short-04/
├── project_status.md       # 项目状态报告（含 12 个子任务的进度）
├── client_history.md       # 与客户的历史邮件摘要（最近 5 封）
├── risk_register.xlsx      # 风险登记表（8 条风险项）
└── email_template.md       # 公司邮件模板
```

**用户指令序列：**
1. "我需要给客户张总写一封项目周报邮件，用 email_template.md 的格式"
2. "重点说明本周完成的里程碑和下周计划"
3. "风险部分只提 high 级别的，不要吓到客户"
4. "语气要专业但友好，参考 client_history.md 的沟通风格"

- recall："本周完成了哪些里程碑？"

- 只包含 high 级别风险（不含 medium/low）
- 格式符合 email_template.md
- 语气与 client_history.md 一致

**度量指标：**
- risk_register.xlsx 的输出截断情况（算法 5）
- 总 token 消耗

### SHORT-05：数据表清洗

**场景：** 清洗一份包含错误和重复的客户数据表。

**测试算法：** 算法 5（回压——大量数据输出的控制）、算法 1（渐进式披露）

**种子数据：**
```
eval_seeds/short-05/
├── customers_raw.csv       # 原始客户数据（500 行，含重复、格式错误、空值）
├── cleaning_rules.md       # 清洗规则（手机号格式、地址标准化、去重规则）
└── region_mapping.yaml     # 区域编码映射表
```

数据质量问题（程序化注入）：
- 15% 的行有重复（基于手机号判定）
- 10% 的手机号格式错误（多位/少位/含字母）
- 8% 的地址缺少区域编码
- 5% 的客户名有前后空格

**用户指令序列：**
1. "读取 customers_raw.csv，按 cleaning_rules.md 的规则清洗"
2. "告诉我发现了多少问题，分类统计"
3. "用 region_mapping.yaml 补全缺失的区域编码"
4. "去重后保存为 customers_clean.csv"
5. "生成一份清洗报告 cleaning_report.md，列出所有操作和统计数据"

**Probe 问题：**
- recall："原始数据中有多少行？清洗后剩多少行？"
- artifact："cleaning_report.md 中列出了哪些清洗操作？"
- decision："去重时以什么字段为判定依据？"

**成功标准：**
- `customers_clean.csv` 无重复行
- 手机号全部符合格式
- 缺失区域编码已补全
- `cleaning_report.md` 包含各类问题的统计数字

**度量指标：**
- CSV 输出截断率（算法 5——500 行数据读取时是否被中间截断）
- 工具调用次数

### SHORT-06：日程冲突检测

**场景：** 检查一周的日程安排中是否存在时间冲突并提出调整建议。

**测试算法：** 算法 4（缓存布局）、算法 5（回压）

**用户指令序列：**
1. "检查这周日程有没有时间冲突"

- recall："检测到几对冲突？分别是哪些事件？"

- 满足 constraints.md 的所有约束

### MID-01：跨文档信息汇总

**场景：** 从 8 份不同格式的文档中提取信息，汇总为一份项目进展报告。

### 种子数据生成规范

所有种子数据程序化生成（参考 Context-Bench [31] 的方法），确保：

1. **虚构实体**：所有人名、公司名、项目名均为虚构，防止模型依赖训练数据
2. **确定性答案**：每个 probe 问题都有程序化验证的 ground truth
3. **可控噪声**：干扰信息（如 SHORT-01 的闲聊、MID-05 的 red herrings）比例可配置
4. **版本锁定**：种子数据有 hash 校验，确保可重复

### Probe 评分流程

```
1. 执行任务 → 收集 Agent 输出
2. 注入 probe 问题 → 用同一 session 继续对话
3. 收集 probe 回答
4. 用独立 LLM（不同模型）盲测评分
5. 6 维度各 0-5 分，取平均
6. 与 ground truth 对比（精确匹配的额外 +1 bonus）
```

### 统计显著性要求

- 每个 case × 每个 config 跑 3 次
- 比较 Full vs Baseline 使用配对 t 检验（paired t-test）
- 消融实验使用 Dunnett 检验（多重比较校正）
- 显著性阈值：p < 0.05
- 效应量报告：Cohen's d

### ACON 配对分析流程（参考 [12]）

```
1. 找到 Full 成功但 Baseline 失败的 case
2. 对比两者的执行轨迹
3. 识别 Plugin 的具体干预点（哪个算法起了作用）
4. 反向：找到 Full 失败但 Baseline 成功的 case
5. 分析 Plugin 是否引入了负面干扰
6. 据此调整算法参数
```

### 5.4 Enron Email Corpus + EnronQA

**来源：** Enron 公司邮件（2001 年公开），EnronQA 为 2025 年基于该邮件集构建的 QA 基准（arXiv:2505.00263）
**规模：** 60 万封真实企业邮件（158 名员工）+ 52.8 万个问答对（150 个用户收件箱，103,638 封邮件）
**获取：**
- Enron 原始邮件: https://www.kaggle.com/datasets/wcukierski/enron-email-dataset
- EnronQA: https://huggingface.co/datasets/MichaelR207/enron_qa_0922
- 清洗版: https://huggingface.co/datasets/LLM-PBE/enron-email

**数据内容：** 真实企业邮件线程，包含转发链、签名块、+1 回复、CC/BCC、附件引用等生产环境噪声。EnronQA 为每个用户收件箱生成了问答对，中位数每用户 1,000+ 个问题，质量经过评估管线验证。关键优势：模型的参数知识不太可能记住 Enron 邮件内容，检索增益可以真实反映在 benchmark 分数上。

**Case 映射：**

| Case | 复用方式 | 说明 |
|------|---------|------|
| SHORT-04 邮件草稿 | 复用真实邮件线程作为 `client_history.md` | 真实沟通风格 |
| MID-02 邮件线程分析 | **最高价值匹配** | 从 Enron 选取 40 封同一线程的邮件 |
| LONG-03 谈判记录 | 参考邮件线程中的多方讨论模式 | Enron 有大量商务谈判邮件 |

**使用建议：** MID-02（多人邮件线程分析）可以直接从 Enron 数据中选取一个真实的多人讨论线程。EnronQA 的问答对可以直接作为 probe 问题的 ground truth——不需要手工编写"第 3 封邮件说了什么"这类问题。

选取邮件线程的脚本思路：

```python
# 从 Enron 数据中筛选适合 MID-02 的邮件线程
# 条件：同一线程 ≥ 30 封邮件、≥ 5 个参与者、跨 2 周以上
import pandas as pd

emails = pd.read_csv("enron_emails.csv")
# 按 Subject (去掉 Re:/Fwd:) 分组
emails["thread"] = emails["Subject"].str.replace(r"^(Re:|Fwd:)\s*", "", regex=True)
threads = emails.groupby("thread").agg(
    count=("Message-ID", "count"),
    participants=("From", "nunique"),
    span_days=("Date", lambda x: (x.max() - x.min()).days)
)
candidates = threads[
    (threads["count"] >= 30) &
    (threads["participants"] >= 5) &
    (threads["span_days"] >= 14)
]
print(f"找到 {len(candidates)} 个候选线程")
```

**数据内容：** 所有实体名称和关系均为虚构（防止模型靠训练数据作弊），问题由 SQL 查询→自然语言转换生成，答案可程序化验证。LLM Agent 使用 `open_files`（读取完整文件）和 `grep_files`（搜索模式）两个工具完成任务。评估维度：搜索查询构造能力、文件操作链接能力、工具选择准确度、数据导航效率、成本效率。

**Case 映射：**

| Case | 复用方式 | 说明 |
|------|---------|------|
| SHORT-03 审批查询 | 复用多文件检索模式 | 跨多份制度文档的查询 |
| SHORT-09 知识问答 | 复用多文档推理任务 | 从文档库中定位答案 |
| MID-01 跨文档汇总 | 复用跨文件关系追踪 | 多来源信息整合 |

**使用建议：** Context-Bench 最大的价值不是数据本身，而是**数据生成方法**。它的 SQL→虚构实体→半结构化文本 管线可以直接复用到我们的种子数据生成器中。具体做法：

```
1. 定义 schema（如 employees、departments、projects 表）
2. 用 SQL 插入虚构数据（Faker 库生成姓名、日期、金额）
3. 用 LLM 将结构化数据转换为自然语言文档（会议纪要、邮件、报告）
4. 从 SQL 查询生成 probe 问题和 ground truth
```

这个管线可以覆盖所有需要"虚构组织数据"的 case，特别是 LONG-04（绩效评估，30 人）和 LONG-08（ISO 认证，114 个控制项）。

### 5.8 实施路线图

**Phase 1：快速启动（1-2 天）**

下载 OfficeBench + SpreadsheetBench，覆盖 SHORT-01~10 中的 8 个 case。这两个数据集格式最接近我们的 case 定义，改动最小。同时下载 EnronQA 作为 MID-02 的种子。

```bash
# Phase 1 下载脚本
pip install huggingface_hub --break-system-packages

# OfficeBench
git clone https://github.com/zlwang-cs/OfficeBench.git eval/datasets/officebench

# SpreadsheetBench
huggingface-cli download KAKA22/SpreadsheetBench --repo-type dataset --local-dir eval/datasets/spreadsheetbench

# EnronQA
huggingface-cli download MichaelR207/enron_qa_0922 --repo-type dataset --local-dir eval/datasets/enronqa
```

**Phase 2：中等任务覆盖（2-3 天）**

下载 OdysseyBench，用 HomerAgents 框架为 MID-01~10 生成长步骤任务变体。HomerAgents 的"对话合成"能力可以自动生成用户指令序列，替代手工编写 30-50 轮指令。

```bash
# Phase 2
git clone https://github.com/microsoft/OdysseyBench.git eval/datasets/odysseybench
# 用 HomerAgents 为 MID 任务生成变体
cd eval/datasets/odysseybench && python homer_agents/generate.py \
  --task-type multi-document \
  --num-steps 30 \
  --output ../seeds/mid-01/
```

**Phase 3：长任务与特殊场景（3-5 天）**

参考 Context-Bench 的 SQL→虚构实体 管线，编写种子生成器。用 Finch 的 Enron 电子表格作为 LONG-07 的真实数据基底。

```bash
# Phase 3 种子生成器骨架
python eval/scripts/generate_seeds.py \
  --case LONG-04 \
  --method context-bench-sql \
  --schema eval/schemas/performance_review.sql \
  --num-employees 30 \
  --output eval/seeds/long-04/
```

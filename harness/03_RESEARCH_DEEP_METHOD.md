# 03_RESEARCH_DEEP_METHOD: 单要素深度研究方法论

> 适用范围：对 012_SYSTEM 中每个 Category (C1-C11) 进行独立深度研究。
> 与 03_RESEARCH_METHOD 的关系：03_RESEARCH_METHOD 解决"发现要素"（归纳），本文解决"理解要素"（分析）。
> 前者回答"Harness 有哪些要素"，本文回答"每个要素的知识结构是什么"。

---

## 一、研究目标

对每个 Category 建立完整的知识图谱，回答六个核心问题：

| # | 问题 | 对应产出 |
|---|------|----------|
| Q1 | 这个问题的**理论基础**是什么？ | 理论溯源：哪些学科/领域早已研究过这个问题 |
| Q2 | 业界实践基于哪些**前提假设**？ | 假设清单：哪些条件成立时方案才有效 |
| Q2.5 | 哪些假设是**硬核**、哪些是**保护带**？ | 假设分级：根基性假设 vs 可替换假设 |
| Q3 | 有哪些**核心算法/策略**？ | 算法-策略谱系：从学术到工程的方案演进 |
| Q4 | 业界**做了哪些实践**？ | 实践案例：谁、做了什么、在什么条件下 |
| Q5 | 取得了**什么效果**？ | 量化结果：指标、数据、对比基线 |
| Q6 | 效果是**如何验证的**？ | 验证方法：实验设计、评估指标、可复现性 |
| Q6.5 | 什么实验结果会**推翻**当前结论？ | 证伪条件：结论的可靠边界在哪里 |
| Q7 | 代码/commit 中隐含了哪些**未写明的设计决策**？ | 隐性知识逆向：从实现反推意图 |

**逻辑链**：Q1(理论) → Q2(假设) → Q2.5(假设分级) → Q3(方案) → Q4(实践) → Q5(效果) → Q6(验证) → Q6.5(证伪) → Q7(隐性知识)

这九个问题构成一条从"为什么可以这样做"到"怎么证明做对了"再到"什么时候会失效"的完整证据链。

### 认识论基础

本框架综合了三种认识论传统，各补充不同的分析视角：

**Lakatos 科学研究纲领（Q2.5 的理论来源）**：Lakatos 将知识分为**硬核**（不可放弃的核心假设）和**保护带**（可调整的辅助假设）。硬核假设被推翻意味着整个方案失效，保护带假设被推翻只需调整策略。Q2 识别了有哪些假设，Q2.5 进一步区分其层级——这决定了方案的脆弱点在哪里。例如 C1 的硬核可能是"上下文窗口有限且利用率影响质量"，保护带是"40% 这个具体阈值"。

**Popper 证伪主义（Q6.5 的理论来源）**：Q6 问"如何证实效果"，Q6.5 问对称问题"什么会推翻结论"。Popper 的立场是：科学命题的价值不在于被证实多少次，而在于能否被证伪。提前想清楚证伪条件，能帮助判断现有结论的可靠边界。例如"40% 利用率阈值"——如果在不同模型上测试发现阈值在 30%-60% 间波动，那"40%"的普适性就被证伪了。

**Nonaka SECI 模型（Q7 的理论来源）**：Q1-Q6 主要覆盖显性知识（论文、数据、算法描述），但 Harness Engineering 中存在大量**隐性知识**——设计决策的 rationale 往往没有写在文档里，只存在于工程师经验和代码变更历史中。Q7 通过对开源代码和 commit history 的逆向分析，将隐性知识外化为可分析的显性知识。

## 二、研究方法：层叠式文献分析法

### 2.1 方法选择理由

单个 Category 的深度研究本质是一个**受控的文献综述**（Scoped Literature Review）。区别于系统性综述（Systematic Review）的全面性要求，我们的研究有明确的起点（012_SYSTEM 中已有的机制描述）和明确的终点（回答六个核心问题）。

采用"层叠式"策略的原因：每个 Category 的知识不是平面的，而是分层的。表层是工程实践（博客、文档），中层是设计模式（架构文献），底层是理论基础（学术论文、经典著作）。从表层向底层挖掘，每一层的发现决定下一层的搜索方向。

### 2.2 三层搜索模型

```
Layer 3（底层）: 理论基础 ─── 学术论文、经典著作、跨学科理论
    ↑ 溯源
Layer 2（中层）: 设计模式 ─── 架构文献、技术标准、算法描述
    ↑ 抽象
Layer 1（表层）: 工程实践 ─── 产品文档、技术博客、开源代码、基准测试
```

**操作流程**：

**Step 1：从 012_SYSTEM 提取锚点**
- 取出该 Category 的所有机制点和引用编号
- 每个机制点提取关键词，构成初始搜索词表
- 标记已有数据（012_SYSTEM 中已列出的量化结论）

**Step 2：Layer 1 扩展——工程实践搜索**
- 搜索渠道：02_REF.md 已有文献深读 → 文献中的引用追踪 → 针对性互联网搜索
- 搜索目标：找到更多实践案例、量化数据、实现细节
- 记录模板：每条实践记录为 `[谁] + [做了什么] + [在什么条件下] + [效果如何] + [出处]`

**Step 3：Layer 2 抽象——设计模式提炼**
- 对 Layer 1 的实践进行归类，识别重复出现的模式
- 搜索该模式是否有已知的架构描述（如 GoF 模式、EAI 模式、微服务模式等）
- 识别模式之间的选择条件（什么场景用什么策略）

**Step 4：Layer 3 溯源——理论基础追踪**
- 对 Layer 2 的设计模式追问：这个方法的数学/理论依据是什么？
- 搜索渠道：学术论文（arXiv、ACL、NeurIPS）、经典教材、跨学科著作
- 重点关注：信息论、认知科学、控制论、分布式系统理论中的已有结论

## 三、分析框架：六维知识卡片

每个 Category 的研究成果组织为六个维度，每个维度对应一个核心问题：

### 维度 1：理论溯源（Theoretical Foundations）→ 回答 Q1

**分析要求**：
- 该问题在 AI Agent 之前是否已存在？在哪些领域被研究过？
- 相关的数学定理、信息论结论、认知科学发现是什么？
- 这些理论为工程实践提供了什么样的约束或指导？

**归纳方法**：跨领域类比归纳——从多个学科中找到研究同一本质问题的不同表述，提取共性结论。

**输出格式**：
```
理论名称 | 来源学科 | 核心结论 | 对 Harness 的启示 | 出处
```

### 维度 2：前提假设（Premises & Assumptions）→ 回答 Q2

**分析要求**：
- 当前工程实践隐含了哪些假设？（例如："上下文窗口有限"是显式假设，"用户会审核输出"是隐式假设）
- 这些假设在什么条件下成立？什么条件下失效？
- 假设失效时，方案会如何退化？

**归纳方法**：反事实分析——对每个实践反问"如果 X 不成立会怎样"，从实践中反向提取假设。

**输出格式**：
```
假设内容 | 显式/隐式 | 成立条件 | 失效后果 | 依赖该假设的机制
```

### 维度 2.5：假设分级——硬核与保护带（Hard Core vs Protective Belt）→ 回答 Q2.5

**理论来源**：Lakatos 科学研究纲领方法论（1970）

**分析要求**：
- 将 Q2 识别的假设分为两级：硬核（推翻则方案失效）和保护带（推翻则需调整策略但框架不变）
- 评估每个硬核假设的稳定性：在技术发展趋势下是否可能被推翻？
- 当保护带假设被推翻时，方案如何调整？是否有已知的调整路径？

**归纳方法**：反事实影响分析——对每个假设构造"如果此假设不成立"的场景，按影响范围判定层级。影响整个 Category 方案论基础的为硬核，仅影响具体参数/策略选择的为保护带。

**输出格式**：
```
假设内容 | 层级(硬核/保护带) | 稳定性评估 | 若失效的影响范围 | 已知的调整路径
```

---

### 维度 3：核心算法与策略（Core Algorithms & Strategies）→ 回答 Q3

**分析要求**：
- 解决该问题的已知算法有哪些？从简单到复杂排列。
- 算法之间的关系是什么？（演进、互补、互斥）
- 每种算法的时间/空间/质量权衡是什么？

**归纳方法**：谱系构建——按时间线或复杂度递进排列算法，识别演进路径和分叉点。

**输出格式**：
```
算法/策略名 | 核心思路 | 复杂度 | 适用场景 | 局限性 | 出处
```

### 维度 4：实践案例（Industry Practices）→ 回答 Q4

**分析要求**：
- 哪些产品/团队实现了该 Category 的哪些机制？
- 实现细节是什么？（架构、代码层面）
- 不同实现之间的差异原因是什么？（场景不同 or 设计哲学不同）

**归纳方法**：多案例比较——对齐多个实践案例的相同维度，通过差异分析识别场景因素。

**输出格式**：
```
实践方 | 具体实现 | 技术栈 | 设计选择理由 | 出处
```

### 维度 5：效果与数据（Results & Evidence）→ 回答 Q5

**分析要求**：
- 有哪些量化数据？指标是什么？基线是什么？
- 数据的来源和可信度如何？
- 不同实践的效果之间是否可比？差异原因是什么？

**归纳方法**：证据汇总——按指标维度汇总所有可用数据，标注置信度。

**输出格式**：
```
指标 | 基线值 | 优化后值 | 提升幅度 | 实验条件 | 出处 | 置信度
```

**置信度分级**：
- A：经同行评审的实验数据（学术论文）
- B：可复现的基准测试数据（开源评估框架）
- C：厂商自报数据（技术博客，有具体数字）
- D：定性描述（"显著提升"，无具体数字）

### 维度 6：验证方法（Validation Methods）→ 回答 Q6

**分析要求**：
- 效果是如何测量的？评估方法本身可靠吗？
- 实验设计是否有对照组？是否控制了变量？
- 结论的泛化性如何？在其他场景下是否验证过？

**归纳方法**：方法论审计——对每个实验/评估的设计进行方法论层面的分析。

**输出格式**：
```
验证方法 | 评估指标 | 对照设计 | 样本量/范围 | 可复现性 | 局限性 | 出处
```

### 维度 6.5：证伪条件（Falsification Conditions）→ 回答 Q6.5

**理论来源**：Popper 证伪主义（1934）

**分析要求**：
- 对每个关键结论，构造证伪命题：什么实验结果出现时，该结论必须被修正或放弃？
- 现有数据是否已经部分触发某些证伪条件？（即现有结论的边界在哪里？）
- 区分"强证伪"（核心结论被推翻）和"弱证伪"（参数/阈值需修正但方向不变）

**归纳方法**：逆向构造——从结论出发，逐一反转其依赖的前提，判断哪些反转会导致结论失效。

**输出格式**：
```
结论 | 证伪条件 | 证伪强度(强/弱) | 当前是否有反面证据 | 结论的可靠边界
```

**示例**：
```
"40%利用率阈值" | 在3+主流模型上测得阈值差异>20pp | 弱证伪（方向不变但参数不普适）| 仅Anthropic单一来源，未跨模型验证 | 可能是Claude特定值
```

### 维度 7：隐性知识逆向（Tacit Knowledge Reverse Engineering）→ 回答 Q7

**理论来源**：Nonaka SECI 模型（1995）——隐性知识 → 显性知识的"外化"过程

**分析要求**：
- 分析主要开源实现（Claude Code、Codex CLI、OpenCode 等）的代码结构和关键 commit
- 从实现选择中推断未写明的设计决策和权衡
- 识别"代码里做了但文档没说"的模式——这些往往是踩坑后的经验结晶

**分析方法**：
1. **代码考古**：找到该 Category 相关功能的核心代码，追踪其演变历史
2. **决策推断**：对比"可以这样做但没这样做"的路径，推断选择理由
3. **模式提取**：从多个实现的共性选择中提取隐含共识

**输出格式**：
```
实现选择 | 被放弃的替代方案 | 推断的决策理由 | 证据来源(代码/commit/issue) | 置信度
```

**注意**：Q7 的产出标记为 `[推导]` 而非 `[事实]`，因为设计意图是从代码逆向推断的，除非有配套的设计文档或 commit message 明确说明。

## 四、推导规则

研究过程中需要从已有事实推导观点。依据项目约束（不允许使用直觉），所有推导必须明确标注方法：

### 4.1 允许的推导方法

| 方法 | 定义 | 示例 | 标注格式 |
|------|------|------|----------|
| 演绎推理 | 从一般原理推导具体结论 | "信息论证明压缩有损 → 上下文压缩必然丢失信息" | `[演绎: 前提A + 前提B → 结论]` |
| 归纳推理 | 从多个案例提取共性 | "Claude/Codex/Cursor 都用文件系统存记忆 → 文件系统是当前主流记忆后端" | `[归纳: 案例1+案例2+...→ 共性结论, N=样本数]` |
| 类比推理 | 从已知领域推导未知领域 | "操作系统的虚拟内存 ≈ Agent 的分层记忆" | `[类比: A领域的X ≈ B领域的Y, 因为共享属性P]` |
| 反证法 | 通过排除反面证明正面 | "如果不做上下文压缩 → 窗口溢出 → 任务失败 → 所以压缩是必要的" | `[反证: 假设非P → 矛盾Q → 所以P]` |

### 4.2 禁止的推导方式

- ❌ 直觉（"感觉应该这样"）
- ❌ 权威论证（"X 公司这么做了所以是对的"）——必须分析为什么
- ❌ 未标注的因果推断（"A 之后发生了 B" ≠ "A 导致了 B"）

### 4.3 观点分级标记

| 标记 | 含义 |
|------|------|
| `[事实]` | 有直接引用支撑的陈述 |
| `[推导]` | 从事实出发经逻辑推导得到，附推导过程 |
| `[假说]` | 合理但尚未验证的推测，标注待验证条件 |
| `[开放问题]` | 目前无法回答，标注所需信息 |

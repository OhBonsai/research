# 研究方法论选择：Harness 核心要素提炼与系统描述

## 一、问题定义

我们面对的研究对象——Harness Engineering——具有以下特征：

1. **多源散布**：定义分散在 LangChain 博客、OpenAI 报告、Anthropic 实践、学术论文、开源项目中，尚无统一的形式化描述
2. **实践先于理论**：它从工程实践（Claude Code、Codex、OpenCode）中涌现，而非从理论推导出发
3. **多维度交织**：涉及记忆、上下文、工具、编排、验证、演化等维度，且各维度之间存在耦合
4. **场景依赖**：当前定义强绑定 Coding Agent 场景，需要迁移到 Office 场景

因此，我们需要的研究方法必须能够：
- 从散布的实践经验中**归纳**出核心要素（自下而上）
- 用结构化框架**描述**这些要素及其关系（系统化）
- 支持**迁移推演**：判断哪些要素场景无关、哪些需要变换（可泛化）
- 提供**评估标准**：判断提炼结果的完备性和一致性（可验证）

## 二、候选方法评估

### 方法 1：扎根理论（Grounded Theory）— 用于要素发现

**来源**：Glaser & Strauss (1967)，Strauss & Corbin 发展的编码范式

**核心机制**：
- **开放编码（Open Coding）**：将所有原始材料（020/021/022 三篇内部文章 + 互联网 30+ 篇参考文献）逐段拆解，标注概念标签。例如：Claude Code 的 /compact 命令 → 标签"上下文压缩"；AGENTS.md 的渐进式披露 → 标签"信息分层加载"
- **轴心编码（Axial Coding）**：将开放编码产生的标签按因果、条件、策略、结果等关系进行聚类，形成"类别"。例如："上下文压缩"+"信息分层加载"+"工具结果卸载" → 类别"上下文生命周期管理"
- **选择性编码（Selective Coding）**：识别核心类别（Core Category），围绕它整合所有类别形成理论。例如：核心类别可能是"有限上下文窗口下的可靠性工程"

**适用性评估**：
- ✅ 天然适合从散布实践中归纳要素（自下而上）
- ✅ 编码过程可追溯、可重复
- ✅ 产出的"类别-属性-维度"结构可直接作为 Harness 要素框架
- ⚠️ 需要对"理论饱和"做判断——何时停止编码
- ⚠️ 结果受编码者主观性影响，需要交叉验证

**推荐用途**：作为**第一步**，从 30+ 篇资料中提炼 Harness 的要素清单

### 方法 2：Zachman 框架 — 用于要素分类本体

**来源**：John Zachman (1987)，企业架构分类框架

**核心机制**：
6×6 矩阵，纵轴为抽象层级（范围/业务模型/系统模型/技术模型/详细表示/运行实例），横轴为六个基本疑问（What/How/Where/Who/When/Why）。每个单元格定义一种特定的架构制品。

**对 Harness 的映射**：

| | What（数据） | How（功能） | Where（网络） | Who（人） | When（时间） | Why（动机） |
|---|---|---|---|---|---|---|
| **范围** | 记忆类型 | 能力清单 | 执行环境 | 角色定义 | 生命周期 | 设计原则 |
| **业务模型** | 知识结构 | 工作流 | 部署架构 | 权限模型 | 触发机制 | 约束规则 |
| **系统模型** | 数据模型 | Agent Loop | 沙箱拓扑 | 审批流 | 调度逻辑 | 验证条件 |
| **技术模型** | 文件格式 | 工具API | 容器配置 | Token管理 | 压缩策略 | 评分器 |

**适用性评估**：
- ✅ 提供完备的分类维度，防止遗漏
- ✅ 强制从多个视角审视同一概念
- ✅ 天然支持"场景迁移"分析——同一个单元格在 Coding 和 Office 场景下的不同填充
- ⚠️ 36 个单元格可能过于细碎，需要裁剪
- ⚠️ 不描述要素间的动态关系

**推荐用途**：作为**分类检验工具**，确保扎根理论提炼出的要素没有结构性遗漏

### 方法 3：一般形态分析（General Morphological Analysis, Zwicky Box）— 用于设计空间探索

**来源**：Fritz Zwicky (1940s)，加州理工学院

**核心机制**：
1. 定义问题的关键参数（维度）
2. 为每个参数列出所有可能的取值（形态）
3. 构建参数×取值的矩阵（形态箱/Zwicky Box）
4. 通过交叉一致性分析（Cross-Consistency Assessment, CCA）排除不可行的组合
5. 剩余组合构成可行的设计空间

**对 Harness 的映射示例**：

| 参数 | 取值 1 | 取值 2 | 取值 3 | 取值 4 |
|---|---|---|---|---|
| 记忆架构 | 文件系统 | 数据库 | 向量存储 | 混合 |
| 上下文策略 | 全量加载 | 渐进披露 | 按需检索 | 分层压缩 |
| 验证机制 | 无验证 | 规则检查 | 测试套件 | 人工审批 |
| 编排模式 | 单Agent | 协调者-执行者 | 流水线 | 对等协作 |
| 隔离级别 | 无隔离 | 会话隔离 | 沙箱隔离 | 工作区隔离 |

**适用性评估**：
- ✅ 系统性地暴露设计空间的全部可能性
- ✅ CCA 交叉一致性分析可排除矛盾组合
- ✅ 天然支持"变与不变"分析——哪些参数在场景迁移时取值变化，哪些不变
- ⚠️ 参数选择本身需要先行的要素提炼
- ⚠️ 组合爆炸问题——5个参数各4个取值就有1024种组合

**推荐用途**：作为**设计空间探索工具**，在场景迁移分析（Coding→Office）时系统枚举可能的 Harness 配置

### 方法 4：可行系统模型（Viable System Model, VSM）— 用于系统结构描述

**来源**：Stafford Beer (1972)，管理控制论

**核心机制**：
任何可行系统（能自主存活于变化环境中的系统）都具有五个递归嵌套的子系统：
- **S1（操作）**：执行基本工作的操作单元 → 对应 Agent 的工具执行层
- **S2（协调）**：处理操作单元之间的冲突 → 对应上下文隔离与子代理协调
- **S3（控制）**：优化整体效能，分配资源 → 对应编排层（协调者角色）
- **S3*（审计）**：不定期深入检查操作实际 → 对应验证与反馈闭环
- **S4（智能）**：扫描外部环境的威胁和机会 → 对应知识更新与自演化机制
- **S5（政策）**：定义系统身份和价值观 → 对应 AGENTS.md 和设计原则

**关键性质**：Ashby 的必要多样性定律（Requisite Variety）——控制系统的多样性必须匹配被控系统的多样性。对 Harness 的启示：Harness 的约束复杂度必须与 Agent 可能行为的复杂度匹配。

**适用性评估**：
- ✅ 提供了一个验证 Harness 结构完备性的理论框架
- ✅ 递归性：VSM 的每个 S1 本身也是一个完整的可行系统 → 天然支持子代理的递归描述
- ✅ 必要多样性定律为"Harness 应该多复杂"提供了理论判据
- ✅ 已有参考文献将 Harness 与控制论关联（腾讯新闻文章）
- ⚠️ VSM 偏重组织结构，不直接描述信息流
- ⚠️ 需要将抽象的 S1-S5 映射到具体的 Harness 组件

**推荐用途**：作为**结构完备性验证框架**，检验提炼出的 Harness 要素是否覆盖了可行系统的全部必要功能

### 方法 5：设计科学研究（Design Science Research, DSR）— 用于整体研究方法论

**来源**：Hevner et al. (2004)，信息系统研究

**核心机制**：
DSR 的核心循环：问题识别 → 制品设计 → 制品评估 → 知识贡献。七条准则：
1. 必须产出可用制品（构念/模型/方法/实例）
2. 面向重要且相关的业务问题
3. 制品的效用必须通过严格评估验证
4. 必须提供清晰可验证的研究贡献
5. 研究方法必须严谨
6. 设计是一个搜索过程
7. 研究必须被有效传播

**适用性评估**：
- ✅ 我们的研究本质上就是一个 DSR 过程——设计一个"企业办公 Agent 的 Harness 架构"制品
- ✅ 提供了制品评估的系统框架
- ✅ "问题相关性"准则要求研究必须解决真实业务问题（10万云桌面客户的办公场景）
- ⚠️ DSR 是上层方法论，需要嵌入具体的分析工具

**推荐用途**：作为**整体研究框架**，确保研究过程的学术严谨性和实践相关性

### 方法 6：ISO/IEC/IEEE 42010 架构描述 — 用于形式化表达

**来源**：ISO/IEC/IEEE 42010:2022

**核心机制**：
标准化的架构描述框架：利益相关者（Stakeholder）→ 关注点（Concern）→ 视角（Viewpoint）→ 视图（View）→ 模型（Model）。强制要求架构描述覆盖所有利益相关者的关注点，通过视角-视图映射确保完备性。

**适用性评估**：
- ✅ 提供了国际标准的架构描述语言
- ✅ 利益相关者驱动——对企业级产品尤其重要
- ⚠️ 较重，适合最终产品架构描述，不适合早期要素探索
- ⚠️ 假设读者熟悉 ISO 术语体系

**推荐用途**：作为**最终产出的表达规范**，在研究后期用标准化语言描述 Harness 架构

## 三、推荐的组合研究方法

经过对上述六种方法的评估，推荐以下组合使用路径：

```
阶段 1: 要素发现（扎根理论）
   ↓ 产出：Harness 概念标签清单 + 类别层级
阶段 2: 结构检验（Zachman × VSM 双重验证）
   ↓ 产出：完备的要素分类体系 + 结构功能映射
阶段 3: 设计空间（形态分析 / Zwicky Box）
   ↓ 产出：Harness 参数空间 + 场景迁移的变与不变
阶段 4: 架构描述（ISO 42010 规范化表达）
   ↓ 产出：面向企业办公场景的 Harness 架构描述文档
```

**整体方法论**：设计科学研究（DSR）覆盖全过程，确保每个阶段的制品都经过评估。

### 各阶段方法的互补关系

| 关系 | 说明 |
|---|---|
| 扎根理论 → Zachman | 扎根理论产出的"类别"需要在 Zachman 矩阵中定位，检验是否存在空白单元格（遗漏的视角） |
| 扎根理论 → VSM | 扎根理论产出的"类别"需要映射到 VSM 的 S1-S5，检验是否覆盖了可行系统的全部功能 |
| Zachman + VSM → 形态分析 | 验证后的完备要素清单作为形态分析的"参数"输入 |
| 形态分析 → ISO 42010 | 形态分析的可行组合作为架构候选方案，用 ISO 42010 规范化描述 |

## 四、具体操作指引

### 阶段 1 操作：扎根理论编码

**输入材料**：项目内部三篇文章（020/021/022）+ 互联网核心文献（02_REF.md 中标★的高价值文章优先）

**开放编码操作步骤**：
1. 逐段阅读每篇文章，对每个具体的工程实践、设计决策、技术组件标注概念标签
2. 标签格式：`[来源编号]-[段落]-[标签名]`，例如 `[021-§4]-协调者不写代码`
3. 同时记录"备忘录"（Memo）：为什么这个概念重要？它解决什么问题？

**轴心编码操作步骤**：
1. 将开放编码的标签按 Strauss & Corbin 范式模型分组：
   - 因果条件（是什么导致了这个实践？）
   - 现象（这个实践本身是什么？）
   - 脉络（在什么条件下发生？）
   - 中介条件（什么促进或阻碍了它？）
   - 行动/互动策略（具体做了什么？）
   - 结果（产生了什么效果？）
2. 聚类形成二级类别

**选择性编码操作步骤**：
1. 从轴心编码的类别中识别核心类别
2. 围绕核心类别构建"故事线"（Storyline）——一段描述 Harness 工程本质的叙事
3. 验证所有类别是否能被核心类别统摄

### 阶段 2 操作：双重结构检验

**Zachman 检验**：
将扎根理论产出的类别填入 Zachman 6×6 矩阵，标记：
- 🟢 已覆盖：类别能明确对应某个单元格
- 🟡 部分覆盖：类别触及但未深入
- 🔴 空白：没有任何类别对应此单元格

空白单元格 = 研究的潜在盲区，需要回到原始材料补充编码或标注为"开放问题"。

**VSM 检验**：
将类别映射到 S1-S5 + S3*，检验：
- 每个子系统是否都有对应的 Harness 组件
- 子系统间的信息通道是否被识别
- 是否满足必要多样性——约束的复杂度是否匹配行为的复杂度

### 阶段 3 操作：形态分析

1. 将验证后的要素作为参数维度
2. 为每个参数列出在 Coding 场景和 Office 场景下的不同取值
3. 用 CCA 排除矛盾组合
4. 标记"不变项"（两个场景取值相同的参数）和"变换项"（取值不同的参数）

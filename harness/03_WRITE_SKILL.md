# 03_WRITE_SKILL: 技术博客写作方法论

> 本文档梳理科技/计算机技术博客的写作手法，供写作前选型。
> 目标读者画像：内部技术论坛，具备专业背景，但不一定了解 Harness Engineering 这一新兴领域。
> 核心挑战：012_SYSTEM 过于精炼（参考手册风格），需要转化为"可阅读的叙事"。

## 零、作者视角素材（用于行文中的个人叙事锚点）

### 素材A：PPT问题的顿悟时刻
- **背景**：领导问"做好一个PPT，除了换更好的模型，还有什么方法？"
- **作者的回答**：先让大模型写HTML，再将HTML转化成PPT
- **顿悟**：后来发现这个方法本质上就是 Harness——不换模型，通过改变模型工作的"环境"（输出格式、中间表示、转换管道）来提升最终质量
- **写作用途**：可作为开篇钩子或核心命题的引入——从一个真实工作场景切入，让读者感到"这不是学术概念，是我们每天都在做的事"

### 素材B：程序员焦虑与职业转型
- **背景**：AI取代编程的时代焦虑——程序员被替代，所有被AI冲击行业的人都很焦虑
- **作者观点**：原本的程序员需要学习的，就是这些 Harness 工程技能
- **历史类比链**：
  - 工业革命：织布工人消失 → 织布机工程师诞生
  - 电力革命：烛台匠消失 → 电气工程师诞生
  - 计算革命：打字员消失 → 软件工程师诞生
  - **AI革命：编程人员转型 → Harness Engineering 专家诞生**
- **核心论点**：每次技术革命消灭的是"直接操作者"，催生的是"系统设计者"。Harness Engineering 就是AI时代的"系统设计"技能
- **写作用途**：可用于文章结尾的"行动号召"或中段的"为什么你应该关心这个"——直击读者（技术人员）的职业焦虑，将学术概念转化为个人发展路径

### 素材使用建议
- 素材A适合放在**文章前1/4**：建立"这跟我有关"的连接
- 素材B适合放在**文章后1/4**：将知识转化为行动动机，制造"读完要去学"的冲动
- 两个素材形成**首尾呼应**：开头"一个PPT问题引发的发现" → 结尾"这不只是一个PPT技巧，这是一个时代的技能迁移"

## 一、行文结构模式（选一）

### S1：金字塔结构（Pyramid Principle）
- **模式**：结论先行 → 分论点 → 支撑证据
- **优点**：高效，读者可在任意深度停止阅读，对忙碌的专业读者友好
- **缺点**：缺少发现感，容易变成"教科书"
- **适用**：读者时间有限、需要快速获取信息的场景

### S2：问题-张力-解决（Problem-Tension-Resolution）
- **模式**：提出实际问题 → 展示现有方案的不足（制造张力）→ 引出新概念作为解决方案
- **优点**：有叙事弧线，读者能感受到"为什么需要这个东西"
- **缺点**：如果问题不痛，张力就假
- **适用**：介绍新概念/新范式（如 Harness Engineering）

### S3：渐进揭示（Progressive Disclosure）
- **模式**：从一个具体场景出发 → 逐层展开抽象 → 最后回到全景
- **优点**：认知负荷低，读者在每一层都有"落脚点"
- **缺点**：节奏较慢，高级读者可能不耐烦
- **适用**：概念层次多、体系复杂的主题

### S4：对比驱动（Before/After Contrast）
- **模式**：每个知识点都通过"没有X会怎样 → 有了X变成怎样"来展开
- **优点**：价值感强，读者立刻明白每个机制的意义
- **缺点**：结构重复，11个要素全用会疲劳
- **适用**：功能/机制类介绍

### S5：混合结构（推荐）
- **模式**：S2（全文主线）+ S3（复杂概念）+ S4（关键要素）
- **全文开篇用 Problem-Tension-Resolution 建立动机
- 要素讲解按复杂度选择 S3 或 S4
- 关键数据点用对比展示冲击力

## 二、知识密度控制手法（多选）

### D1：概念分层引入
- **做法**：每引入一个新概念，先用一句话给出"日常直觉版"定义，再给技术精确版
- **示例**：先说"Harness 就是给 AI 搭的脚手架"，再展开精确定义
- **效果**：读者先建立模糊但正确的心智模型，再逐步精化

### D2：类比锚定（Analogy Anchoring）
- **做法**：将新概念锚定到读者已知的技术概念上
- **012_SYSTEM 已有**：CPU/RAM/OS/App 类比
- **可扩展**：每个 C 类别配一个技术类比（如 C4 验证管道 ≈ CI/CD Pipeline）
- **注意**：类比必须在关键属性上成立，否则会误导。每个类比需标注"类比边界"

### D3：数据冲击点（Data Punch）
- **做法**：在概念阐述后，用一个反直觉的数据打破预期
- **012_SYSTEM 天然优势**：有大量量化数据（6.7%→68.3%，52.8%→66.5%）
- **手法**：数据前先铺设直觉预期，让反差最大化
- **示例**："你可能以为换个更强的模型能解决问题——但同一个 Opus 4.6 在不同 Harness 下排名相差 28 位"

### D4：呼吸节奏（Pacing）
- **做法**：高密度段落后跟一个低密度段落（总结/类比/过渡），形成"吸-呼"节奏
- **经验法则**：连续3个以上机制点后必须插入一个消化段
- **消化段类型**：小结句、过渡问句、回顾类比、"这意味着…"段

### D5：信号词标记
- **做法**：用显式信号词帮助读者定位自己在文章中的位置
- **类型**：
  - 预告："下面我们看三个关键机制"
  - 转折："但这还不够——"
  - 层级："在此基础上"
  - 回顾："回到我们开头的问题"

## 三、故事与案例插入策略（多选）

### N1：开篇场景钩子（Opening Hook）
- **位置**：文章开头
- **做法**：用一个具体的工程困境开场——让读者在3句话内产生"我也遇到过"的共鸣
- **示例方向**：一个 Agent 在第47步开始"胡言乱语"的真实场景
- **长度**：3-5句，不超过一段

### N2：要素级微案例（Per-Element Mini Case）
- **位置**：每个 C 类别的开头或结尾
- **做法**：用一个 2-3 句话的具体场景说明这个机制为什么存在
- **示例**："团队给 Agent 塞了完整的代码库文档——上下文窗口吃满后，它连简单的函数签名都记不住了"（引出 C1）
- **注意**：不需要每个 C 都有，选 3-4 个关键的即可

### N3：贯穿案例线（Running Example）
- **位置**：全文贯穿
- **做法**：选一个具体任务（如"让 Agent 完成一个跨模块重构"），每个要素用这同一个任务来举例
- **优点**：读者只需建立一次场景心智模型
- **缺点**：不是所有要素都适合同一个场景
- **变体**：主线用一个案例，个别要素用独立微案例补充

### N4：反模式警示（Anti-Pattern Stories）
- **位置**：要素讲解中穿插
- **做法**：描述"不这么做会怎样"的灾难场景
- **012_SYSTEM 已有**：上下文死亡螺旋、100%完成但33%合规
- **效果**：比正面说教更有说服力

### N5：行业数据叙事（Data Storytelling）
- **位置**：文章中后段或各章节数据点处
- **做法**：不是简单列数据，而是用叙事包装——"当 LangChain 团队只优化 Harness 而不更换模型时，他们看到了…"
- **效果**：数据从"冷数字"变成"有主语的故事"

## 四、降低阅读难度手法（多选）

### E1：前置知识显式声明
- **做法**：文章开头列出"本文假设你了解：LLM基础概念、Agent基本架构、Prompt Engineering基础"
- **效果**：读者自我评估是否需要先补课

### E2：概念地图先行
- **做法**：在正文前提供一张 11 要素全景图（简化版），让读者先建立框架
- **形式**：一张图 + 一句话描述每个 C，不超过半页
- **效果**：后续阅读每个要素时，读者始终知道"我在地图的哪个位置"

### E3：分级阅读路径
- **做法**：正文中标记 [核心] 和 [进阶] 内容，急于求快的读者只看 [核心]
- **适用**：内容确实有不同深度层次时
- **注意**：不要过度标记，否则视觉噪音

### E4：摘要盒子（Summary Box）
- **做法**：每个 C 类别结尾放一个"一句话总结"盒子
- **格式**：> **C1 一句话**：上下文窗口就是 Agent 的工作记忆——塞太满它就会忘事。
- **效果**：扫读友好，复习友好

### E5：术语渐进定义
- **做法**：首次出现的术语在行内给出简短定义（括号注释或破折号解释），不依赖独立术语表
- **示例**："Harness（模型之外的一切——提示词、工具、执行环境、编排逻辑）"
- **效果**：读者不需要跳转查阅

### E6：视觉节奏调节
- **做法**：长文本段落间插入结构化元素（图表、代码块、对比表、引用块）打破视觉单调
- **经验**：每500字至少一个非纯文本元素
- **注意**：图表必须服务于理解，不做装饰

## 五、技术博客结构模板（供参考）

```
1. 标题（简洁有力，可含核心观点）
2. 摘要/导语（3-5句，读完知道文章讲什么、为什么重要）
3. 开篇钩子（场景/问题/反直觉数据）
4. 背景与动机（为什么需要 Harness Engineering）
5. 核心概念（Agent = Model + Harness + 类比框架）
6. 要素体系（11要素，按逻辑分组而非编号平铺）
   - 分组建议：
     a. 基础层：C1上下文 + C2记忆（Agent的"工作空间"）
     b. 约束层：C3约束 + C4验证（"护栏"）
     c. 协作层：C5编排 + C8人机协作（"谁做什么"）
     d. 进化层：C6自演化 + C7可拆卸（"持续适应"）
     e. 保障层：C9可观测 + C10安全 + C11评估（"看得见、管得住、测得准"）
7. 关系与全景（要素间如何协作，可用图）
8. 数据说话（关键量化证据，集中冲击或分散嵌入均可）
9. 落地启示/行动建议（对读者的直接价值）
10. 总结（回应开篇，形成闭环）
```

## 六、写作偏好选型（已确认）

参考风格标杆：阮一峰（技术科普清晰直白）、酷壳陈皓（工程视角+犀利观点）、品玩/极客公园的深度技术报道（叙事驱动+数据支撑）。选型原则：逻辑连贯、易懂、去AI味。

| 维度 | 选择 | 选择理由 |
|------|------|----------|
| 行文结构 | **S2 问题-张力-解决** | Harness是新概念，读者首先需要理解"为什么需要它"。S2的"痛点→不足→方案"弧线天然适合新范式布道。不用S5混合——混合结构容易显得散，S2一条主线到底更有力 |
| 知识密度控制 | **D1 + D3 + D4** | D1概念分层（先直觉后精确）解决术语门槛；D3数据冲击点是012_SYSTEM的天然优势，不用白费；D4呼吸节奏防止读者窒息。**不选D2**（类比锚定）——012已有CPU/RAM/OS类比，够用了，再加会显得刻意。**不选D5**（信号词）——信号词用多了是典型AI写作味道 |
| 故事插入策略 | **N1 + N4 + N5** | N1开篇钩子用素材A（PPT问题）；N4反模式穿插在关键要素中（上下文死亡螺旋、100%完成33%合规等012已有的好素材）；N5数据叙事让冷数字有主语。**不选N2/N3**——11个要素每个配微案例太重，贯穿案例线又会限制叙事自由度 |
| 降低阅读难度 | **E2 + E4 + E5** | E2概念地图先行——11要素体系必须有一张全景图让读者心里有数；E4摘要盒子——每组要素结尾一句话总结，方便扫读；E5术语渐进定义——行内解释，不跳转。**不选E1**（前置知识声明）——内部论坛读者自己会判断。**不选E3**（分级标记）——增加视觉噪音，博客不是教材。**不选E6**——视觉节奏通过E2/E4自然形成 |
| 要素展开方式 | **分组展开（5组）** | 11个逐一展开太像参考手册（012_SYSTEM的问题本身）；只讲5-6个又不完整。分5组，每组有一个统一主题，读者记住5个"层"比11个编号容易得多 |
| 文章长度预期 | **5000-6000字** | 3000字压不住11要素+个人叙事；8000+在论坛里劝退。5000-6000字大约15分钟阅读时间，是技术深度文的甜区 |
| 语言风格 | **工程实战** | 严谨学术太端着，轻松对话显得不专业。工程实战风格=说人话但有干货：观点明确、例子具体、不绕弯子、不堆形容词。句子短，段落短，一段一个点 |

### 去AI味的具体约束

1. **禁用词表**：不用"深入探讨"、"值得注意的是"、"不难发现"、"众所周知"、"毫无疑问"、"让我们"、"接下来我们将"
2. **段落控制**：每段不超过4句话。宁可多分段，不要一段说三件事
3. **观点前置**：每段第一句就是观点或结论，后面是支撑。不要"首先...其次...最后"的八股排比
4. **个人语气**：允许"我"、"我们团队"、"我发现"——有主语的句子比无主语的被动句更像人写的
5. **不对称结构**：11个要素的展开篇幅故意不均匀——重点要素多写，次要要素少写。AI倾向于给每个点相同篇幅，人不会
6. **口语化转折**：用"但是"代替"然而"，用"说白了"代替"换言之"，用"问题在于"代替"需要指出的是"
7. **数据嵌入叙事**：不单独列数据表（那是012的做法），把数据嵌进句子里——"LangChain团队不换模型只改Harness，成绩从52.8%涨到66.5%"

### 最终文章骨架预览

```
标题：不换模型，换环境——Harness Engineering 的11个工程支点

开篇（素材A）：一个PPT问题引发的思考
  ↓ 张力：为什么换了最贵的模型，Agent还是不靠谱？
  ↓
核心概念：Agent = Model + Harness
  ↓ 全景图（E2）
  ↓
五组要素（S2主线贯穿）：
  第一组「工作空间」：C1上下文 + C2记忆
  第二组「护栏」：C3约束 + C4验证
  第三组「分工」：C5编排 + C8人机协作
  第四组「进化」：C6自演化 + C7可拆卸
  第五组「保障」：C9可观测 + C10安全 + C11评估
  ↓
关系全景 + 关键数据（N5叙事化）
  ↓
结尾（素材B）：从织布工到织布机工程师
```

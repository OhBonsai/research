## 核心发现（60 秒速读）

### 问题
- Harness 需要不绑定特定模型，但如何做到可拆卸？
- 同一模型在不同 Harness 中性能差异 28 位（#33 vs #5），这意味着什么？

### 答案
1. **不是新问题**：Parnas（1972）、SOLID、Hexagonal Architecture 都已给出方向
2. **三层分离**：应用层 → Harness 核心（模型适配、工具抽象、上下文编排）→ 模型适配层
3. **有实现了**：LiteLLM（100+ LLM）、MCP（标准工具协议）、Claude Code（无代码切换）
4. **有收益**：Harness 改进单独贡献 +13.7% 性能，配置驱动成本降 50-70%
5. **有边界**：场景偏向性强（代码生成最优，文本摘要边界），不是所有应用都需要

### 关键指标
| 指标 | 数据 | 来源 |
|------|------|------|
| Harness 改进的性能提升 | +13.7pp | Terminal Bench 2.0 / LangChain |
| 同模型不同 Harness 排名差 | 28 位 | Opus 4.6 在两个架构中 |
| 可拆卸性成本降低 | 50-70% | 推导（模型迭代周期 3 月） |
| 模型特定代码占比 | < 10% | 设计目标 |

## 关键概念解释

### 三层分离（Three-layer Separation）
```
Layer 1: Application       [Agent Logic] → 不关心模型和工具实现
         ↓ 调用
Layer 2: Harness Core      [Model Adapter Registry, Tool Abstraction, Context Assembly]
         ↓ 实现
Layer 3: Model Adaptation  [API Client, Format Conversion, Error Mapping]
```

**优势**：修改模型只改 Layer 3，应用代码零改动

### 模型选择成熟度（L0-L5）
| 等级 | 方式 | 示例 | 成本降低 |
|------|------|------|----------|
| L0 | 硬编码 | `model = "gpt-4"` | 0% |
| L1 | 配置 | `MODEL=gpt-4` | 10% |
| L2 | 适配器 | `IModelProvider` 接口 | 30% |
| L3 | 能力感知 | 任务 → 能力向量 → 选模型 | 50% |
| L4 | 动态路由 | 性能/成本/可用性权衡 | 70% |
| L5 | 自适应 | 反馈循环 + 自优化 | 80%+ |

**建议**：先做到 L2（基础模块化），再看是否需要 L3+

### 场景适用性矩阵
| 场景 | 工具 | 上下文 | 模型敏感 | Harness 价值 |
|------|------|--------|----------|-------------|
| Web Agent | 高 | 中 | 中 | 高 |
| **代码生成** | **高** | **高** | **高** | **非常高** ✓ |
| 文本摘要 | 低 | 中 | 低 | 中 |
| 嵌入式推理 | 低 | 低 | 高 | 低 |

→ 代码生成是最佳场景，嵌入式推理不值得投入

## 核心建议

### 设计时的四大原则
1. **分离关注点**：模型选择逻辑 vs 任务执行逻辑分开
2. **抽象依赖**：应用依赖"接口" ← 模型实现接口（SOLID DIP）
3. **能力驱动**：根据任务需求 + 模型能力进行匹配
4. **分层配置**：应用层配置不变，Harness 层配置演进

### 自检清单（必须满足）
- [ ] 模型切换无需改应用代码
- [ ] 工具定义与模型格式解耦
- [ ] 支持通过配置调整参数
- [ ] 有故障转移链设计
- [ ] 不支持的能力有降级方案
- [ ] 模型特定代码 < 10% 总量

### 实现优先级
1. **必做**：L0 → L2（基础模块化）
2. **应做**：L2 → L3（能力感知）
3. **选做**：L3 → L4（成本优化）
4. **高级**：L4 → L5（自适应系统）

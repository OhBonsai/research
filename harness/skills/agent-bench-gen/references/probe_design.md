# Probe 问题设计指南

## 4 种 Probe 类型

### 1. Recall（事实性保留）

测试 Agent 是否记住了早期输入的具体事实。

**设计原则：**
- 问具体的数字、名称、日期（不是模糊的"大概"）
- 优先问跨压缩边界的早期信息（最容易丢失）
- 涉及个体数据时，问两个不同个体的差异（测是否混淆）

**示例库：**
| 场景 | Probe | 为什么好 |
|------|-------|---------|
| 项目排期 | "关键路径总工期多少天？" | 精确数字，有确定答案 |
| 绩效评估 | "emp_07 和 emp_15 的互评差异？" | 测两个个体是否混淆 |
| 邮件分析 | "第3封和第15封邮件间的立场变化？" | 跨压缩边界 |
| 谈判记录 | "会议3中对方的价格报价？" | 早期数据，容易被压缩丢失 |

### 2. Artifact（工件追踪）

测试 Agent 是否正确追踪了生成的文件和数据。

**设计原则：**
- 问"生成了什么"，而不是"应该生成什么"
- 验证文件内容与过程数据的一致性
- 关注文件间的交叉引用

**示例库：**
| 场景 | Probe | 为什么好 |
|------|-------|---------|
| 月报生成 | "报告中的数字与源文档一致吗？列 3 个" | 交叉验证 |
| 合同审查 | "checklist 15 项中几项通过？" | 统计一致性 |
| ISO 认证 | "差距报告中几个控制项'不合规'？" | 计数一致性 |

### 3. Plan（任务规划）

测试 Agent 是否能基于上下文制定合理的下一步计划。

**设计原则：**
- 问"如果 X 变了，怎么调整？"（条件规划）
- 问"下一步该做什么？"（序列规划）
- 答案应与已收集的信息逻辑一致

**示例库：**
| 场景 | Probe | 为什么好 |
|------|-------|---------|
| 项目排期 | "新增紧急需求应插入哪里？" | 需要理解依赖关系 |
| 文档审阅 | "第二轮审阅应重点关注什么？" | 需要综合第一轮发现 |
| 谈判策略 | "对方突然强硬，应对方案？" | 需要理解谈判态势 |

### 4. Decision（推理链保留）

测试 Agent 是否记住了做出关键决策的理由。

**设计原则：**
- 问"为什么选了 A 而不是 B？"
- 问"排除了什么？基于什么证据？"
- 答案必须有证据链，不是泛泛而谈

**示例库：**
| 场景 | Probe | 为什么好 |
|------|-------|---------|
| Bug 分析 | "排除了哪些误导线索？为什么？" | 需要回溯排查过程 |
| 竞品分析 | "为什么建议聚焦竞品 C？" | 需要数据支撑的决策 |
| 预算编制 | "为什么削减部门 B 的差旅 20%？" | 需要历史数据+政策依据 |

## Probe 覆盖矩阵

每个 case 至少覆盖 2 种 Probe 类型。推荐覆盖策略：

| Case 类型 | 推荐 Probe 组合 | 理由 |
|----------|----------------|------|
| SHORT（5-10 轮） | recall + artifact | 短任务重点测基本信息保留 |
| MID（20-50 轮） | recall + decision + artifact | 中等任务开始涉及推理 |
| LONG（100+ 轮） | 全部 4 种 | 长任务全面测试信息保留 |

## 压缩边界 Probe

长任务中最有价值的 Probe 是跨压缩边界的。设计时：

1. 预估压缩发生的轮次（通常每 30-50 轮触发一次）
2. 在压缩点之前的轮次植入关键信息
3. 在压缩点之后的轮次提问

```
轮 1-30: 输入 12 个 Bug 报告    ← 信息植入
轮 31:   触发第一次压缩          ← 压缩边界
轮 35:   "BUG-103 的症状是什么？" ← 跨边界 recall probe
```
